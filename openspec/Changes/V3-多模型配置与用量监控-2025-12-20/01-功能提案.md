# V3 变更包（盘根 · AI 指南针）功能提案：多模型配置与用量监控

## 1. 背景

当前系统的 AI 能力主要依赖单一模型服务配置，存在以下维护与运营问题：

- AI 能力无法覆盖国内主流大模型厂商生态，导致可选模型范围受限。
- 模型服务的 Base URL 与 API Key 缺乏统一的后台管理入口，无法形成标准化配置流程。
- 模型服务配置缺少权限边界，存在敏感信息被前端侧持久化、被接口返回、被日志记录的风险。
- 缺少按用户维度的用量数据，无法判断实际请求量、稳定性、错误率、延迟、成本估算等运营指标。
- 未来接入新的模型厂商时，如果没有统一的适配层与契约，将引入大量重复改动并扩大回归风险。

本变更包定义一套“可扩展多模型引擎 + 后台统一配置 + 用量监控”的目标规格，为后续施工提供唯一图纸。

## 2. 用户故事

### 2.1 管理员（后台运营者）

- 作为管理员，我在后台选择模型厂商，填写 Base URL，填写 API Key，保存后即可为全站生效。
- 作为管理员，我可以为每个厂商配置默认模型名称，以及可用模型列表。
- 作为管理员，我可以启用、停用某个厂商配置，并设置当前全站默认使用的厂商配置。
- 作为管理员，我可以对某个厂商配置执行连通性测试，得到明确的成功、失败原因与耗时。
- 作为管理员，我可以轮换 API Key，旧 Key 立即失效，新 Key 立即生效，并且配置变更产生审计记录。
- 作为管理员，我可以查看用量监控面板，按时间范围查看：请求次数、成功率、错误率、平均延迟、Tokens 统计、成本估算。
- 作为管理员，我可以按用户、厂商、模型、功能模块进行筛选，并查看明细请求记录。
- 作为管理员，我可以导出用量明细用于对账与运营分析。

### 2.2 普通用户（产品使用者）

- 作为普通用户，我在不需要填写任何 Key 的前提下正常使用 AI 搜索、方案生成等能力。
- 作为普通用户，我可以在前台选择要使用的模型厂商与模型，系统按该选择发起请求。
- 作为匿名用户，我在登录前的模型选择与用量记录在登录后自动绑定到账号。
- 作为普通用户，我的请求在系统侧被记账，管理员可以在后台看到我的使用情况。
- 作为普通用户，当模型服务不可用时，我依然可以获得演示模式结果，页面展示规则保持一致。

### 2.3 研发维护者（工程维护者）

- 作为研发维护者，我可以以“新增 Provider Adapter”的方式接入新厂商，不需要修改业务 UI。
- 作为研发维护者，我可以在统一的 AI 契约下返回结果，避免每个厂商返回结构不同导致前端分支爆炸。
- 作为研发维护者，我可以在日志中定位某一次请求的厂商、模型、耗时、错误类别，并且敏感信息被脱敏。

## 3. 目标

### 3.1 支持的模型厂商范围（首批）

系统必须支持以下厂商配置：

- 智谱（GLM）
- MiniMax
- 通义千问
- Gemini
- OpenAI

系统必须支持后续新增厂商，新增厂商不修改既有业务 UI 代码。

### 3.2 后台管理能力

- 提供后台“模型服务配置”管理页
- 支持厂商配置的新增、编辑、启用、停用、删除
- 支持 Base URL 配置
- 支持 API Key 配置（加密存储、接口脱敏返回、界面遮罩展示）
- 支持默认模型设置
- 支持连通性测试
- 支持 Key 轮换并记录审计日志

### 3.3 用量监控能力

- 对每一次 AI 请求记录用量事件
- 用量事件必须包含：用户标识、厂商、模型、功能模块、时间、耗时、状态、错误类别、Tokens 统计、成本估算
- 提供后台“用量监控”面板：概览指标、趋势图、分组统计、明细列表、导出

### 3.4 安全与合规

- API Key 禁止出现在前端构建产物
- API Key 禁止被任意 API 返回
- API Key 禁止被日志系统记录
- 管理后台能力必须具备明确的权限控制边界
- 配置变更必须具备审计日志

## 4. 非目标

以下内容不在本变更包范围内：

- 会员计费、套餐计费、按次计费、自动扣费等商业闭环
- 多租户企业级权限体系、SSO、组织架构管理
- 对接第三方监控平台的完整指标管道
- 针对每个用户单独存储自带 Key 的能力

## 5. 约束

- 本变更包仅产出规格文档，不进入代码施工。
- 所有设计必须符合“敏感信息最小暴露原则”，任何页面与接口不得呈现 Key 明文。
- 多模型扩展机制必须以“配置驱动 + Adapter 接口”的方式实现，不引入散落在各处的厂商分支。
