# V3 变更包（盘根 · AI 指南针）技术方案：多模型配置与用量监控

## 1. 方案总览

本方案引入“AI 网关服务”作为统一出入口，承担以下职责：

- 后台管理：模型厂商配置、Base URL、API Key、默认模型、启用状态
- 调用适配：对接不同厂商的 HTTP API，屏蔽差异，统一输出业务契约
- 安全控制：Key 加密存储、脱敏展示、权限校验、审计记录
- 用量监控：记录每次请求的指标数据，提供聚合查询与导出

前端应用不再直接持有真实 Key，不再直接请求第三方模型厂商 API。所有 AI 请求统一调用 AI 网关服务。

## 3. 领域模型与数据结构

### 3.1 模型厂商枚举

厂商标识 `vendor` 采用枚举字符串：

- `zhipu`
- `minimax`
- `qwen`
- `gemini`
- `openai`

### 3.2 厂商配置（ProviderConfig）

#### 3.2.1 字段

- `id`：字符串，全局唯一
- `vendor`：字符串，取值见 3.1
- `displayName`：字符串，后台展示名
- `baseUrl`：字符串，厂商 API 基础地址
- `apiKeyEncrypted`：字符串，加密后的 Key
- `apiKeyMask`：字符串，脱敏展示文本，仅用于后台列表展示
- `defaultModel`：字符串，默认模型名称
- `models`：数组，模型列表，元素为 `ModelConfig`
- `isEnabled`：布尔值，是否启用
- `isDefault`：布尔值，是否为全站默认
- `createdAt`：时间
- `updatedAt`：时间
- `createdBy`：字符串，管理员用户 id
- `updatedBy`：字符串，管理员用户 id

#### 3.2.2 规则

- 任意时刻 `isDefault = true` 的记录数量必须为 1。
- `apiKeyEncrypted` 永不下发前端。
- 后端对外返回配置时，仅返回 `apiKeyMask`。

### 3.3 模型配置（ModelConfig）

- `model`：字符串，厂商模型名称
- `displayName`：字符串，后台展示名
- `inputPricePer1M`：数值，每 100 万输入 Tokens 价格
- `outputPricePer1M`：数值，每 100 万输出 Tokens 价格
- `isEnabled`：布尔值

说明：成本估算以本表为准。

### 3.4 身份与账号（Identity, User）

系统必须同时支持匿名身份与账号身份，用于用量监控与模型选择。

#### 3.4.1 Identity

- `id`：字符串，全局唯一
- `userId`：字符串，允许为空
- `preferredProviderId`：字符串，允许为空
- `preferredModel`：字符串，允许为空
- `createdAt`：时间
- `updatedAt`：时间
- `lastSeenAt`：时间

规则：

- 首次访问时系统生成 Identity，并下发到客户端 cookie。
- 后续请求通过 cookie 携带 Identity 标识。
- 登录成功时服务端将当前 Identity 绑定到账号，并写入 `userId`。
- 绑定完成后，历史 UsageEvent 通过 Identity 聚合到账号维度展示。

#### 3.4.2 User

- `id`：字符串
- `email`：字符串
- `passwordHash`：字符串
- `role`：字符串，取值 `user`、`admin`
- `status`：字符串，取值 `active`、`disabled`
- `createdAt`、`updatedAt`

### 3.5 用量事件（UsageEvent）

#### 3.5.1 字段

- `id`：字符串
- `identityId`：字符串
- `userId`：字符串，允许为空
- `feature`：字符串，取值：`ai_search`、`solution_generate`
- `providerId`：字符串
- `vendor`：字符串
- `model`：字符串
- `requestId`：字符串，请求链路唯一标识
- `startedAt`：时间
- `finishedAt`：时间
- `latencyMs`：数值
- `status`：字符串，取值：`success`、`failed`
- `errorCategory`：字符串，取值：`network_error`、`auth_error`、`rate_limit`、`timeout`、`parse_error`、`unknown`
- `inputTokens`：数值
- `outputTokens`：数值
- `totalTokens`：数值
- `estimatedCost`：数值

#### 3.5.2 规则

- 默认不存储原始提示词与原始输出文本。
- 允许存储 `promptSize` 与 `responseSize` 作为调试指标。

### 3.6 审计日志（AuditLog）

- `id`
- `actorUserId`
- `action`：字符串，取值：`provider_create`、`provider_update`、`provider_delete`、`provider_enable`、`provider_disable`、`provider_set_default`、`provider_rotate_key`
- `targetId`：字符串，目标 ProviderConfig id
- `createdAt`
- `detail`：JSON，脱敏后的变更摘要

## 4. 后端 API 契约

### 4.1 鉴权

- 前端登录成功后获取 `accessToken`。
- 管理后台接口必须校验：用户登录状态、用户角色为 `admin`。
- 登录成功时服务端必须绑定当前 Identity 到账号，保证历史用量可查询。

### 4.2 运行时 AI 网关接口

#### 4.2.0 可用模型清单

- **Path**：`GET /api/ai/options`
- **Response**：
  - `providers`：数组，元素包含 `providerId`、`vendor`、`displayName`、`models`
  - `defaultProviderId`：字符串

说明：前端用于构建“模型选择”列表。

#### 4.2.1 AI 搜索

- **Path**：`POST /api/ai/search`
- **Request**：
  - `query`：字符串
  - `domain`：字符串
  - `toolIds`：字符串数组
  - `articleIds`：字符串数组
  - `providerId`：字符串，允许为空
  - `model`：字符串，允许为空
- **Response**：返回 `AISearchResultV2`，字段完全遵守既有前端契约：
  - `mode`：`ai`、`demo`
  - `fallbackReason`：允许为空
  - `summary`、`recommendation`、`suggestedTools`、`suggestedArticles`

说明：

- 当 `providerId` 与 `model` 均为空时，服务端读取 Identity 中的 `preferredProviderId` 与 `preferredModel`。
- 当 Identity 中未设置首选配置时，服务端使用全站默认 ProviderConfig。
- 失败时返回演示模式结果，并记录用量事件。

#### 4.2.2 方案生成

- **Path**：`POST /api/ai/solution`
- **Request**：
  - `goal`：字符串
  - `toolIds`：字符串数组
  - `domain`：字符串
  - `providerId`：字符串，允许为空
  - `model`：字符串，允许为空
- **Response**：
  - `mode`：`ai`、`demo`
  - `fallbackReason`：允许为空
  - `solutionMarkdown`：字符串

### 4.3 管理后台接口

#### 4.3.1 列表

- **Path**：`GET /admin/ai/providers`
- **Response**：ProviderConfig 列表，包含 `apiKeyMask`，不包含 `apiKeyEncrypted`

#### 4.3.2 新增

- **Path**：`POST /admin/ai/providers`
- **Request**：
  - `vendor`
  - `displayName`
  - `baseUrl`
  - `apiKeyPlain`：字符串，仅用于提交，服务端接收后立即加密存储
  - `defaultModel`
  - `models`
- **Response**：新建后的 ProviderConfig（脱敏）

#### 4.3.3 编辑

- **Path**：`PUT /admin/ai/providers/{id}`
- **Request**：允许修改 `displayName`、`baseUrl`、`defaultModel`、`models`、`isEnabled`
- **Response**：更新后的 ProviderConfig（脱敏）

#### 4.3.4 轮换 Key

- **Path**：`POST /admin/ai/providers/{id}/rotate-key`
- **Request**：`apiKeyPlain`
- **Response**：更新后的 ProviderConfig（脱敏）

#### 4.3.5 设置默认

- **Path**：`POST /admin/ai/providers/{id}/set-default`
- **Response**：成功状态

#### 4.3.6 连通性测试

- **Path**：`POST /admin/ai/providers/{id}/test`
- **Response**：
  - `status`：`success`、`failed`
  - `latencyMs`
  - `message`

### 4.4 用量监控接口

#### 4.4.1 概览

- **Path**：`GET /admin/ai/usage/overview`
- **Query**：`from`、`to`、`vendor`、`model`、`userId`、`identityId`
- **Response**：
  - `requestCount`
  - `successCount`
  - `failedCount`
  - `avgLatencyMs`
  - `totalTokens`
  - `estimatedCost`

#### 4.4.2 明细

- **Path**：`GET /admin/ai/usage/events`
- **Query**：`page`、`pageSize`、`from`、`to`、`vendor`、`model`、`userId`、`identityId`、`feature`
- **Response**：分页列表

#### 4.4.3 导出

- **Path**：`GET /admin/ai/usage/export`
- **Query**：同 4.4.2
- **Response**：CSV 文件流

## 5. Provider Adapter 设计

### 5.1 统一接口

定义 `ProviderAdapter` 接口：

- 输入：`prompt`、`model`、`baseUrl`、`apiKeyPlain`
- 输出：`{ text, inputTokens, outputTokens }`

### 5.2 扩展机制

- 每个 `vendor` 对应一个 Adapter 文件
- 新增厂商时新增 Adapter 并注册到 Provider Registry
- 业务层只依赖 Registry，不依赖具体厂商

### 5.3 失败处理策略

- 后端调用失败时返回 `mode = demo` 的结果
- `fallbackReason` 使用统一枚举，便于监控统计

## 6. 安全设计

### 6.1 Key 加密

- 后端接收 `apiKeyPlain` 后立即加密
- 数据库仅存储 `apiKeyEncrypted`
- 仅在后端内存中解密使用，解密后只用于本次请求

### 6.2 Key 脱敏

- 后台列表与详情接口仅返回 `apiKeyMask`
- `apiKeyMask` 规则：显示末尾 4 位，其余使用 `*` 遮罩

### 6.3 日志脱敏

- 禁止记录 `apiKeyPlain`、`apiKeyEncrypted`
- 禁止记录完整 prompt 文本
- 仅记录尺寸指标与错误摘要

### 6.4 权限控制

- 管理后台接口必须后端校验管理员角色
- 运行时 AI 网关接口必须校验用户身份

## 7. 管理后台 UI 布局（前端侧目标）

### 7.1 导航结构

- AI 引擎
  - 模型服务配置
  - 用量监控
  - 审计日志

### 7.2 模型服务配置页

- 列表表格字段：展示名、厂商、默认模型、启用状态、是否默认、Base URL、Key 状态（脱敏）、更新时间

### 7.3 用量监控页

- 顶部筛选：时间范围、厂商、模型、用户、功能模块

- 趋势图：按天聚合的请求数、Tokens、成本
- 明细表：时间、用户、厂商、模型、功能、耗时、状态、Tokens、成本
- 导出：导出当前筛选条件下的明细 CSV

## 8. 风险与对策

- **风险：模型厂商 API 差异较大**
  - **对策**：以 Adapter 层隔离差异，Registry 统一选择，业务只依赖统一接口

- **风险：Key 安全**
  - **对策**：加密存储、脱敏返回、日志脱敏、后端权限强校验

- **风险：监控数据量增长**
  - **对策**：按时间分区、按天聚合表、保留策略，后台默认只展示聚合数据
