# V2 变更包（盘根 · AI 指南针）功能提案：可维护性增强

## 1. 背景

当前仓库已完成 V1 施工包交付，核心能力包含：本地存储持久化、AI 搜索回退方案、演示模式提示条、主 PRD 规格回写。

随着后续功能继续迭代，项目需要在不改变用户可见功能的前提下，提升工程可维护性，降低新增需求的实现成本，降低回归风险，降低多人协作的沟通成本。

当前可维护性主要问题清单：

- **单文件体积过大**：`1pangen-ai-compass/App.tsx` 体积约 60KB，包含大量视图 JSX、状态、业务函数、UI 逻辑，导致定位困难与改动风险提升。
- **横切能力分散**：本地存储键名、解析校验、写入副作用、提示条状态目前在入口文件内聚集，不利于复用与单点修正。
- **AI 能力边界不清晰**：UI 端以 `summary` 文案包含特定短语作为演示模式判断依据，属于隐式契约，未来扩展易引入破坏性改动。
- **规格目录不一致**：仓库根目录存在 `openspec/`，应用目录内同时存在 `1pangen-ai-compass/openspec/`，宪法文件目前位于应用目录内，主规格位于仓库根目录内，单一事实来源不够集中。
- **根目录包管理残留**：仓库根目录存在 `package-lock.json`，应用目录内存在独立的 `package-lock.json`，根目录锁文件缺少对应 `package.json`，容易误导维护者。

## 2. 用户故事（User Story）

- **作为**：未来的功能开发者
- **我需要**：在不阅读全部入口文件的前提下，快速定位功能所在模块，并完成安全修改
- **从而**：在相同时间内交付更多功能，并降低回归风险

- **作为**：代码审查者
- **我需要**：在一次改动中明确看到变更边界与依赖影响
- **从而**：以更低成本完成审查并避免隐性缺陷

## 3. 目标（Goals）

本变更包在 V2 达成以下目标：

- **G1 模块边界清晰**：将横切能力从 `App.tsx` 抽离到独立模块（存储、AI、领域过滤），`App.tsx` 仅保留装配与路由级视图选择逻辑。
- **G2 契约显式化**：为 AI 搜索结果增加显式字段表达运行模式与回退原因，UI 不再依赖文案包含关系做判断，同时保持主规格中对文案的硬性要求。
- **G3 规格单一事实来源**：确立仓库根目录 `openspec/` 为唯一规格根目录，将宪法迁移至根目录，并写明目录约束。
- **G4 工程干净**：移除根目录无效锁文件，确保包管理入口唯一且可追踪。

## 4. 非目标（Non-goals）

本变更包不包含以下内容：

- UI 重设计
- 新增业务功能
- 引入后端服务
- 引入账号登录真实闭环

## 5. 约束（Constraints）

- **C1 规格一致性**：不得改变 `openspec/Specs/PRD-盘根AI指南针-最终版.md` 已定义的用户可见行为。
- **C2 原子改动**：执行清单每一项任务的代码改动控制在 50 行以内。
- **C3 不泄露敏感信息**：任何文档与代码不得出现密钥明文。
- **C4 兼容性约束**：不引入额外兼容层，除非该兼容层用于消除重复入口并形成单一事实来源。

## 6. 交付物（Deliverables）

- `openspec/Changes/V2-可维护性-2025-12-20/` 全量文档：
  - `01-功能提案.md`
  - `02-技术方案.md`
  - `03-执行清单.md`
  - `04-规格增量.md`
- 新增模块文件（按执行清单定义）并完成接入
- 主规格与宪法完成同步归档
