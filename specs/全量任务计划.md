# 全量任务计划

> 生成日期：2026-04-25  
> 目标：把校园站、全球站、统一后台与横向平台能力推进到全量 PRD 可验收状态。  
> 基线：当前代码以本地仓库为准；最近已完成提交 `416f41b Unify campus account compliance around platform identity`。  
> 规则：中文文案；早期注册只要求用户名 + 密码，邮箱/手机号后置绑定；不写长期兼容层；UI 图标只用 Lucide；新接口走命名空间；前端不得包含 API Key；cn/com 原始数据与跨域 Cookie/token 不默认共享。

## 0. 总体策略

### 0.1 交付顺序

1. 校园站闭环：先让校园站从“能用”到“可闭环”，包括搜索转求助、通知、空间认领和内容质量。
2. 全球站真实化：先账号准入，再内容模型，再搜索与方案闭环，最后用户后台。
3. 横向平台能力：AI 网关、analytics、notification/email、billing 在两站核心路径落地后统一抽象。
4. 旧路径删除：只删除已有替代接口、已有 smoke 覆盖、已确认无页面依赖的旧路径。
5. 全量验证与发布：最终才做 README、PRD 对照、release checklist、push/tag。

### 0.2 执行原则

- 每个切片完成后必须可运行、可测试、可回滚。
- 共享契约先改 `@ns/shared`，再改 server，最后改前端。
- 涉及数据库 schema 的切片必须包含 `db:push`、`db:seed`、server 测试和至少一个 smoke。
- 涉及前端流程的切片必须包含中文加载态、空态、错误态。
- 涉及后台审核、账号、合规、数据隔离、支付、额度的切片必须补服务端测试。
- 每个提交遵守 Lore Commit Protocol。

### 0.3 当前已完成项

- [x] 平台地基：`SiteContext`、统一 envelope、admin-console、基础模块目录。
- [x] Identity 后端：注册、登录、me、邮箱验证、密码重置、token invalidation。
- [x] Compliance 后端：协议查询、consent、数据导出、注销申请、后台处理注销。
- [x] Moderation 基础：任务创建、查询、状态流转、审计日志。
- [x] Admin Console 基础：登录、站点切换、队列、详情、用户、内容、审计、配置。
- [x] 校园举报切 `/api/moderation/tasks`。

## 10. 风险与缓解

| cn/com 数据隔离回归 | 合规风险高 | 每个跨站接口补 site-aware 测试 |
| 文档与代码状态漂移 | 后续接手误判 | 每个里程碑结束同步 `当前现状.md` 和 release checklist |

### 13.3 总体结论

全量任务计划的主线成立：先补校园闭环，再做全球真实化，再抽横向平台能力，最后删除旧路径和发布。

该顺序符合使命：

- 校园站：AI 无形服务生活，搜索和空间形成“问题 -> 答案 -> 反馈 -> 沉淀”的闭环。
- 全球站：AI 有形赋能，以方案生成和工具/内容推荐帮助用户把目标落地。
- 两站连接：校园站提供自然转化触点，全球站承接主动学习和能力提升。

### 13.4 必须修正的文档冲突

#### 冲突 A：校园账号方式

现象：

- 宪法与校园 PRD 的旧段落仍写“手机号 + 验证码 / SMS”。
- 部分旧计划和交接文档写“用户名 + 邮箱 + 密码”，把邮箱当作注册必选项。

裁定：

- 当前目标架构采用“用户名 + 密码”作为早期最小注册。
- 邮箱、手机号作为后置绑定能力，不作为早期注册必选项。
- 不实现手机号/SMS 注册分支。
- 后续必须修订宪法和 PRD 旧段落，避免未来执行者误判。

影响任务：

- `M0` 增加文档冲突修订。
- `M2` 全球账号仍保留申请制、邀请制、GitHub OAuth。

#### 冲突 B：AI 无结果回答 vs 冷启动诚实策略

现象：

- 校园 PRD 多处描述“本地无结果 -> AI 回答”。
- 宪法要求内容不足时 AI 搜索无结果不得编造，必须展示相关已有内容 + 收集用户需求入口。

裁定：

- AI 可以给“参考性、非事实断言”的整理建议，但不能伪造校园事实。
- 无结果页必须同时提供“发起求助 / 我知道准确信息，来补充”入口。
- 本地人工确认内容的优先级高于 AI 回答。

影响任务：

- `M1-A` 搜索无结果转求助为校园站硬前置。
- `M5-A` AI 网关必须实现敏感词、fallback reason、输出降级和日志。

#### 冲突 C：AI 推荐/精选边界

现象：

- 校园 PRD 有“AI 根据时间/热度实时推荐”的描述。
- 同一 PRD 和宪法又要求 AI 不自动精选/发布，不替人做价值判断。

裁定：

- AI 可做排序建议、摘要和需求分析。
- 首页推荐卡片的最终可见结果必须基于可解释规则：人工确认数、更新时间、搜索缺口、运营配置或审核通过内容。
- 不做黑盒 AI 自动精选。

影响任务：

- `M1-D` 校园内容质量应优先补真实 seed 和可解释推荐规则。
- `M5-B` analytics 用于支撑推荐依据，不替代人工裁决。

#### 冲突 D：AI 额度和限流落点

现象：

- 宪法要求 AI 搜索从第一天起限流，登录用户有明确额度，额度耗尽给提示。
- 全量计划原先把 billing/quota 放到 `M5-D`，过晚。

裁定：

- 完整计费系统可以放在 `M5-D`。
- 但最小 AI 限流/额度提示必须提前成为公开 AI 路径的发布门槛。

影响任务：

- `M0` 增加 AI 资源治理最小门槛。
- `M5-D` 保留完整 billing/quota、订单和手动支付。

#### 冲突 E：账号注销与数据删除深度

现象：

- 当前 compliance 已有注销申请和 token 失效。
- 宪法要求账号注销后 30 天内物理删除。
- 全球 PRD 要求 GDPR 合规。

裁定：

- 当前实现只算最小闭环，不算最终合规完成。
- 后续必须补异步删除任务、删除审计、30 天物理删除验证和跨站隔离验证。

影响任务：

- `M4-C` 全球用户后台必须包含数据导出/删除。
- `M7` 全量验证必须包含数据删除专项。

- [ ] `M0` 必须补 AI 限流/额度提示的最小发布门槛。
