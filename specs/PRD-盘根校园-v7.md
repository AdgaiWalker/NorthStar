# 盘根 — 产品需求文档（PRD）

> 版本：v8.0
> 日期：2026-04-21
> 产品经理：Cat Wu

## 一、一句话

盘根是区域知识基础设施。以高校为起点，为每个城市构建可靠、活的本地知识网络。

---

## 二、核心定位

| 维度      | 说明                                   |
| ------- | ------------------------------------ |
| **是什么** | 区域知识基础设施 — 可靠、有人维护的本地信息系统            |
| **为谁**  | 第一阶段：高校学生；终局：城市全体居民                  |
| **单位**  | 以城市为单位，高校为锚点。一个城市一个站点                |
| **愿景**  | 每个城市都有一个盘根，像自来水和电网一样不可或缺             |
| **路径**  | 校园验证 → 城市扩展。校园跑不通，城市免谈；校园跑通了，城市是自然延伸 |

### 2.1 服务周期

面向大学生的完整四年周期：

| 阶段     | 年级  | 核心需求 | 内容方向            |
| ------ | --- | ---- | --------------- |
| **适应** | 大一  | 生存指南 | 食堂、宿舍、报到、校园周边   |
| **探索** | 大二  | 发现兴趣 | 社团、技能、AI工具、学习方法 |
| **聚焦** | 大三  | 明确方向 | 实习、考证、考研、就业     |
| **交换** | 大四  | 资源变现 | 二手、经验传承、商家对接    |

### 2.1b 扩展路径

校园是 PMF 验证场。验证通过后，模型复制到城市：

| 阶段     | 范围      | 内容              | 验证指标               |
| ------ | ------- | --------------- | ------------------ |
| **v1** | 校园      | 食堂、选课、报到、宿舍、二手  | 学生日活 100+，有人写第二篇文章 |
| **v2** | 校园 + 周边 | 商家入驻、租房、交通、周边生活 | 商家主动申请入驻           |
| **v3** | 城市      | 政务办事、社区服务、城市生活  | 非学生用户占比 >30%       |

关键原则：**每一个场景都要重新验证信任链。** 校园的信任链跑通了，不等于政务的信任链能跑通。不跳步。

### 2.2 与竞品的差异

| 竞品   | 盘根的不同                                    |
| ---- | ---------------------------------------- |
| 小红书  | 小红书是泛化的，盘根是区域化的；小红书信息分散在帖子里，盘根结构化在知识库里   |
| 知乎   | 知乎是问答，盘根是知识库+论坛；知乎不鼓励持续更新，盘根通过使用反馈保持内容活性 |
| 贴吧   | 贴吧是讨论，盘根是结构化知识+经营空间；贴吧内容杂乱，盘根有质量门槛       |
| 飞书文档 | 飞书是团队工具，盘根是公开平台；飞书没有社区，盘根有反馈和发现机制        |

**核心差异：知识库即个人领地。** 不只是写文章，是经营自己的知识空间。

### 2.3 为什么不用微信群

微信群是盘根最直接的竞争对手。学生已经在群里问"食堂几点关门"、"选课怎么选"。

盘根能做到而微信群做不到的三件事：

| 维度     | 微信群                | 盘根                           |
| ------ | ------------------ | ---------------------------- |
| **累积** | 每年新生重复问同样的问题，没人翻历史 | 2024 年的回答在 2026 年还在、还准确、有人维护 |
| **深度** | 一句话问答，无法承载结构化内容    | 文章形态承载完整测评、全流程指南             |
| **经营** | 回答沉了就没了，没人记得是谁说的   | 知识库挂在作者名下，有收藏、有确认、有身份        |

**盘根不是比微信群好，是做微信群做不到的事。**

### 2.4 作者心理学

认证作者为什么要持续在盘根上投入？三层内在驱动力：

**第一层：整理认知（给自己写）**

- "我吃了两年食堂，心里有了一套判断，不写下来就散了"
- 写的过程就是整理自己认知的过程
- 知识库的结构化形态天然迎合"我想把这件事理清楚"的冲动
- 盘根只需要不打扰他

**第二层：被看见（给他人写）**

- 小池塘里的大鱼效应：在黑河学院，"最懂食堂的人"有社交价值
- 微信群里的回答沉了就沉了，盘根上名字挂在知识库上
- 作者需要看到自己的影响力数据："你的报到指南帮助了 234 名新生"

**第三层：留下痕迹（给未来写）**

- 大学四年短暂，很多人毕业时遗憾"搞明白的事走了就没了"
- 认领机制的本质：你的东西会留下来，会有人接着做
- 触发人最深层的需求——不想白活

**核心洞察：作者不离开盘根，不是理性判断，是沉没成本加身份认同。盘根上的那个我，比微信群里那个我更真实。**

### 2.5 信任链

盘根卖的是信任。核心信任链：

```
认证 → 质量保证 → 读者信任 → 作者被需要 → 作者持续投入
  ↑                                              │
  └──────────────────────────────────────────────┘
```

任何破坏信任的事，都不能做。

## 三、核心概念

### 3.1 内容分类体系

帖子、文章是同级内容，知识库是分类容器。全量版本支持两种内容形态，降低认知复杂度。

```
知识库（分类层）── 每个认证作者自动创建一个：
├── 我的知识库  ← 认证后自动创建，帖子+文章统一归属

知识库内包含的同级内容：
├── 帖子（短）  ← 快问快答、短分享、求助、二手、活动通知、讨论（用标签区分类型）
└── 文章（长）  ← 长期凝练的精华，结构化内容
```

**简化原则（公理五·做减法）：**

- 只做帖子 + 文章两种内容形态。话题用"帖子 + #话题标签"代替
- 每个认证作者初始只有一个自动创建的"我的知识库"
- 全量版本支持话题作为独立内容形态，并支持手动创建更多知识库

**关键规则：**

- 知识库是分类容器，一个知识库可以同时包含帖子和文章
- 认证后系统自动创建"我的知识库"，所有帖子和文章归入其中
- 帖子用标签区分用途（#求助 #二手 #活动 #讨论 #分享）

### 3.2 知识库

**知识库是平台的核心单元。** 类似飞书知识库，但开放给所有人。

| 特性  | 说明                                       |
| --- | ---------------------------------------- |
| 所有权 | 每个认证用户自动获得一个"我的知识库"，全量版本支持手动创建更多知识库 |
| 内容  | 知识库内可包含帖子和文章，文章支持文字、图片、表格                |
| 经营性 | 知识库可以是纯知识分享，也可以是个人小店、商家页面          |
| 协作  | 创建者为所有者，可邀请他人共建                      |

### 3.3 帖子

| 特性   | 说明                                      |
| ---- | --------------------------------------- |
| 形式   | 短文本，类似论坛帖子                              |
| 用途   | 快问快答、短分享、求助、二手交易、活动通知、讨论                |
| 标签   | 用 `#求助` `#二手` `#活动` `#讨论` `#分享` 等标签区分用途 |
| 归属   | 帖子默认归入作者的知识库                            |
| 生命周期 | 按时间线展示，滑动加载。热门帖子通过活动排序自然浮上来             |

**帖子 + 标签即可发起讨论：**

- 帖子加 `#讨论` 标签即可发起讨论，其他用户通过回复参与
- 讨论型帖子与普通帖子的区别仅在于标签，交互方式一致
- 全量版本支持独立的话题内容形态，支持讨论主题 + 参与周期

### 3.4 信任等级体系

渐进式信任，参考 Linux.do 的 TL 系统，但适配校园场景。

| 等级      | 名称   | 权限                       | 获得方式         |
| ------- | ---- | ------------------------ | ------------ |
| **Lv0** | 游客   | 阅读、搜索                    | 无需注册         |
| **Lv1** | 注册用户 | 发帖、收藏、反馈"有帮助/有变化"        | 手机号/微信注册     |
| **Lv2** | 活跃用户 | 反馈记录积累、可被认证              | 发帖≥3 且活跃≥7天  |
| **Lv3** | 认证作者 | 写文章、影响力面板          | 平台审核         |
| **Lv4** | 资深作者 | 认领无人维护的知识库、优先审核、协助社区管理   | Lv3 持续贡献≥90天 |
| **管理**  | 管理员  | 内容审核、用户管理、数据统计、认证审批、等级调整 | 团队内部         |

**关键原则：**

- Lv3 不是终点。持续贡献的认证作者可以升到 Lv4，获得认领知识库等高级权限
- 等级升级条件透明，用户知道自己离下一级还差什么
- 等级可降，但降级前必须有提醒和改进路径（公理三·多元世界观）：
  - 降级前 7 天系统通知："你的活跃度正在下降，再发 1 篇帖子即可保持当前等级"
  - 用户清楚知道"我离保持等级还差什么"，偏差是改进信号而非惩罚
  - 降级后可重新积累升级，不是永久失去
- 认证审核权在平台团队，必须人工审核，**不可自动通过**
- 初期团队自己生产种子内容

### 3.5 "+" 按钮行为分化

同一个 "+" 按钮，认证前后展示不同选项：

**未认证用户点击 "+"：**

```
┌─────────────┐
│  💬 发帖子   │  ← 快问快答、短分享、求助
│             │
│  ─────────  │
│  📝 申请认证  │  ← 引导去认证，解锁完整权限
└─────────────┘
```

**认证作者点击 "+"：**

```
┌─────────────┐
│  💬 发帖子   │
│  📝 写文章   │  ← 认证后解锁
└─────────────┘
```

> 认证后自动创建"我的知识库"，用户无需管理多个知识库。全量版本开放手动创建更多知识库。

**设计意图：**

- 降低门槛：任何人都能发帖子，让信息流动起来
- 质量控制：文章需要认证，保证结构化内容的质量
- 自然引导：未认证用户看到"申请认证"入口，明白还有更高权限
- 认证用户只有两个核心动作——发帖子或写文章，减少认知负担

## 四、页面结构

### 4.0 导航

```
桌面端导航栏（固定顶部）：
[logo]  [最新] [知识库] [我的]  [🔍搜索...]  [头像]

移动端底部导航（3 平坦 Tab）：
[首页] [知识库] [我的]
```

**导航原则（公理五·做减法）：**

- 不使用浮动创建按钮或灵动岛。创建入口放在首页 Hero 区域内联，不占用导航位置
- 移动端 3 个 Tab 等宽排列，无突出、无浮动元素
- 桌面端导航不包含 [+创建] 按钮，创建动作通过首页内联入口触发

### 4.1 首页（最新动态）

论坛首页，顶部简洁 Hero（搜索 + 创建），下方混合内容 Feed。



**Hero 区域设计**：

- 无冗余标题。用户知道自己在哪，不需要重复品牌名
- 搜索框占满宽度，placeholder 用具体问题示例引导（如"图书馆几点关门？食堂哪最好吃？"）
- 右下角"写点什么"按钮，点击触发创建流程
- 搜索提示文案："从知识库中查找，找不到再问 AI"

**搜索行为（自动检测，无模式切换）**：

- 用户输入后自动先匹配本地知识库内容（关键词匹配）
- 有本地结果：直接展示匹配列表，无需等待 AI
- 无本地结果：自动调用 AI 回答，AI 回答附带关联文章引用
- 不暴露"关键词模式"和"智能模式"的概念，用户只需搜索

**Feed 卡片设计**：

- Feed 中不显示收藏按钮。收藏功能仅在详情页可用
- 原因：Feed 的任务是浏览和发现，收藏决策在详情页做出
- Feed 卡片信息精简：作者 + 时间 + 内容摘要 + 统计

**关键机制**：

- **活动排序**：默认按最近活动排序（阅读、收藏、反馈都算活动），不是按创建时间
- **滑动加载**：参考抖音机制，无限滚动，带入场动画（卡片渐入）
- 混合展示：帖子 + 文章更新 + 知识库更新，统一时间线

**帖子预览悬浮窗**：

- 点击 Feed 中的帖子卡片，不直接跳转详情页，而是弹出居中预览浮层
- 浮层展示：作者信息、帖子完整内容、图片网格、标签、回复数/浏览数
- 三种关闭方式：点击右上角 X、点击 backdrop 遮罩、按 Escape 键
- 浮层底部提供「查看全部评论」按钮，点击进入完整帖子详情页
- 设计目的：降低用户浏览成本，快速阅览后再决定是否进入详情页互动

### 4.2 知识库页

展示所有知识库的入口。类似论坛版块列表。



**排序规则**：

- 热门：按收藏数+阅读数加权
- 最新创建：按创建时间
- 可按分类筛选

- 父文章和子文章各自拥有独立的 TOC
- 导航树中展开/折叠子文章层级
- 点击子文章时，中栏 TOC 和右栏阅读器同步切换到该子文章

### 4.5 创建入口

**不使用浮动按钮或独立创建页面。创建入口内联在首页 Hero 区域。**

用户点击首页"写点什么"按钮后，根据认证状态弹出不同菜单：

**认证后用户看到的：**

```
┌─────────────┐
│  💬 发帖子   │  ← 短文本，直接发布
│  📝 写文章   │  ← AI 对话式写作
```

**未认证用户看到的：**

```
┌─────────────┐
│  💬 发帖子   │  ← 短文本，直接发布
│             │
│  ─────────  │
│  📝 申请认证  │  ← 引导认证，解锁写文章
└─────────────┘
```

**写文章流程（AI 对话式写作）**：

1. 选择目标知识库（已有或新建）
2. AI 提问："你想分享什么？"用户自由表达
3. AI 通过追问补充细节（价格是人均还是一份？什么时候人最少？）
4. AI 基于用户回答排列内容结构，用户可调整顺序、删除、补充
5. 用户确认后发布

### 4.6 我的

同一页面，根据认证状态显示不同内容。

**未认证用户看到的：**

- 顶部：头像、昵称、学校、"注册用户"标签
- 我的帖子、我的收藏、我的反馈（注册用户即有）
- 认证引导卡（视觉突出）：展示认证后可解锁的能力——写文章、看到影响力数据——底部有"申请认证"按钮
- 设置

**认证作者看到的：**

- 顶部：头像、昵称、学校、"认证作者"标签
- 我的知识库（认证后自动创建，帖子+文章统一归属）
- 我的帖子、我的收藏、我的反馈
- 设置、认证信息
- 影响力面板：帮助人数、文章阅读量、收藏数

**设计意图：** 认证引导卡不是横幅广告，是"你离这些只差一步认证"。未认证用户每次进"我的"都能看到，是最自然的转化入口。认证后引导卡消失，替换为真实的影响力面板和知识库列表。

## 五、内容活性机制

核心理念：**读就是确认，用就是更新。** 让更新发生在使用过程中，而不是作为单独的动作。

参考 Linux.do 的活动排序和渐进信任，但适应盘根的本地知识特性。

### 5.1 反馈机制

文章底部只有一个问题："这篇内容对你有帮助吗？"

- **[ 有帮助 ]** — 一次确认信号，刷新文章的"上次确认"时间
- **[ 有变化 ]** — 弹出一行输入框"什么变了？"，提交后通知作者

没有三选一纠错表单，没有评分，没有颜色标签。就像日常对话——"有用"或者"不对了，我告诉你新的"。

### 5.2 排序逻辑

首页动态流默认按"最近活动"排序，不是按创建时间。以下行为都算"活动"：

- 有人读了文章
- 有人点了"有帮助"
- 有人反馈了"有变化"
- 有人收藏了
- 作者更新了内容

**不需要"推送到最新"按钮。** 阅读本身就是推。没人读的内容自然沉下去，有人读的内容自然浮上来。

### 5.3 时间透明

每篇文章显示"上次确认于 X 天前"。这是事实，不是评价。读者自己判断"30 天前的确认，大概还靠谱"。

不使用 🟢🟡🔴 颜色标签。原因：颜色标签是公开的成绩单，作者看到自己的文章变黄会产生焦虑，伤害信任链。

作者收到"有变化"反馈时，系统私下通知。文章长期无人阅读时（30 天以上），系统私下提醒作者"你的文章 XX 已经一段时间没人看了，需要更新吗？"——不公开，不评价。

### 5.4 认领机制

知识库作者毕业后无人维护时，Lv4 资深作者可以"认领"该知识库继续更新。需要 Lv4 权限（Lv3 持续贡献≥90天）。

### 6.5 人机协同原则

**决策权永远在用户手中。AI 是助手，不是作者。**

盘根所有 AI 功能遵循同一原则：

| AI 的职责                | 用户的权利          |
| --------------------- | -------------- |
| 提问，帮助用户想清楚要说什么        | 写什么、不写什么，由用户决定 |
| 排版，把用户说的话整理成结构        | 先说什么后说什么，由用户调整 |
| 格式化，处理 Markdown 等技术细节 | 最终发布的内容，由用户确认  |
| 搜索和推荐，帮用户找到相关信息       | 看什么、信什么，由用户判断  |

**具体到写作流程：**

1. 用户打开写作页，AI 提问：你想分享什么？
2. 用户自由表达，AI 通过追问帮助补充细节（价格是人均还是一份？什么时候人最少？）
3. AI 基于用户的回答排列内容结构，用户可以调整顺序、删除、补充
4. 用户确认后发布

AI 不代笔。AI 不替用户决定什么重要。AI 的角色是让表达更容易，但内容的主人永远是用户。

## 九、不做的事

### 9.1 产品边界

- 不做评论区 → 帖子用回复，文章用"有帮助/有变化"反馈
- 不做点赞 → 用收藏代替
- 不做粉丝/关注
- 不做泛化内容 → 只做区域内容
- 不做手机端单独设计 → 自适应响应式
- 不做富文本编辑器，用 AI 对话式写作代替传统编辑器，底层 Markdown
- 不做用户生成分类 → 知识库本身就是分类

### 9.2 战略红线

以下事项破坏信任链（见 2.5 节），无论任何阶段、任何增长压力，都不可触碰：

| 红线             | 原因                                                |
| -------------- | ------------------------------------------------- |
| **不做流量生意**     | 排序逻辑从准确度变成热度，作者就会追流量而不是追准确，活性机制死亡                 |
| **不开放知识库创建权限** | 去掉认证门槛，知识库质量崩塌，读者信任断裂。信任链第一环不可砍                   |
| **不做算法推荐**     | 算法优化停留时长，盘根要的是"找到准确答案然后走"。首页 = 时间线 + 搜索           |
| **不为了增长降低标准**  | 宁可 5 个高质量知识库，不要 50 个半成品。空平台不可怕，烂平台才可怕             |
| **不让商业伤害信任**   | 商家内容必须标明身份，"打印店指南"不能是打印店老板的软文。读者必须知道"这个信息是帮我还是卖我" |

## 十、设计语言

### 10.1 视觉基调

- **默认主题**：白色（`#FFFFFF` 背景），干净的纸张感
- 不使用暖色调或纹理感（如奶油色、纸张噪点）。白色 = 干净 = 可信
- 边框和分割线使用极浅灰（`#E5E5E5` / `#EEEEEE`），低存在感
- 强调色使用自然绿色系（Sage Green），与校园自然感呼应
- 圆角统一 12px（卡片）/ 8px（按钮、输入框），现代但不夸张
- 文字层级：标题 600 加粗 / 正文 400 常规 / 辅助文字浅灰，不使用过多字号

### 10.2 删减原则

**最好的设计是不存在的部分。** 每个元素都要回答"不显示它会怎样？"。

- 不显示用户已知的品牌名（首页 Hero 不放标题）
- 不在 Feed 放收藏按钮（收藏决策在详情页）
- 不暴露搜索模式切换（自动检测即可）
- 不使用浮动按钮或灵动岛（增加视觉噪音）
- 动效克制：仅在 Feed 入场使用渐入动画，无装饰性动效

### 10.3 桌面端优先

盘根定位为区域高质量内容平台，桌面端是主力使用场景（阅读长文章、管理知识库），移动端用于快速浏览和发帖。

**桌面端布局原则：**

- 充分利用宽屏：三栏布局（导航树 | TOC | 阅读器）
- 信息密度适中：不像论坛那么密集，也不像移动端那么稀疏
- 参考 Linux.do 的清晰列表结构，但保持盘根自己的知识库特色

## 十一、文章悬浮窗卡片（Skill Card）

### 11.1 概念

悬浮窗是嵌入在文章正文中的引用卡片，用于展示关联内容。类似飞书文档中的嵌入块，但更轻量。

### 11.2 结构

```
┌──────────────────────────────┐
│  📎 卡片标题                   │
│                               │
│  [展开 ▼]                     │
│                               │
│  （点击展开↓）                 │
│  ┌──────────────────────────┐│
│  │ 具体内容：提示词 / 模板 /  ││
│  │ 代码 / 步骤等原始素材      ││
│  └──────────────────────────┘│
└──────────────────────────────┘
```

### 11.3 交互

一个按钮：**展开 / 收起**。点击展开显示内部具体内容（提示词、模板、代码、步骤等原始素材），再次点击收起。

> 公理二（熵减）：两个按钮增加认知复杂度（"效果"和"内容"有什么区别？），一个展开/收起足够。

### 11.4 使用场景

- 作者在文章中引用一个 AI 提示词 → 点击展开提示词原文
- 作者嵌入一个工具/技能卡片 → 点击展开完整介绍
- 作者引用另一个知识库的文章 → 点击展开摘要预览

## 十三、Obsidian 生态集成

### 13.1 战略定位

**Obsidian 是私人第二大脑，盘根是公开第二大脑。** 两者都以 Markdown 为基础、以知识库为结构，天然互补。

### 13.2 分阶段实施

| 功能 | 目标用户 | 说明 |
| ---- | ------ | ---- |
| 格式兼容 | 团队内部 | 渲染器支持 Wikilinks/Callout/Frontmatter |
| Obsidian Vault 导入 | 团队内部 | 文件夹→知识库，文件→文章，嵌套→子文章 |
| Obsidian 社区插件 | 认证作者 | 本地写完一键发布到盘根 |
| 双向同步 | 生态 | 盘根反馈回 Obsidian、鲜度状态同步 |

> **内部工具边界**：Obsidian 集成是团队内部生产效率工具，不作为面向用户的功能宣传。
> 普通用户不需要知道 Obsidian 是什么。格式兼容的目的是让团队能用熟悉的生产工具高效产出种子内容。
> 只有当认证作者主动问"我能用 Obsidian 写吗"时，才开放 Vault 导入。

### 13.3 格式兼容（团队内部）

盘根 Markdown 渲染器需支持以下 Obsidian 语法：

| 语法          | 示例                     | 转换规则             |
| ----------- | ---------------------- | ---------------- |
| Wikilinks   | `[[二食堂测评]]`            | 转为盘根内部文章链接       |
| Callout     | `> [!note] 提示`         | 渲染为盘根 Callout 组件 |
| Frontmatter | `---\ntags: [食堂]\n---` | 解析为文章元数据         |
| 嵌入          | `![[图片.png]]`          | 渲染为图片            |
| 标签          | `#食堂 #测评`              | 映射为盘根分类标签        |

### 13.4 Vault 导入（团队内部工具）

```
Obsidian Vault 结构          盘根结构
─────────────────          ──────────
📁 校园美食/          →     📚 校园美食（知识库）
  📄 麻辣烫测评.md     →     📝 麻辣烫测评（文章）
  📄 一食堂.md         →     📝 一食堂（文章）
  📁 二食堂/           →     📂 二食堂（子文章组）
    📄 麻辣烫.md       →     📝 麻辣烫（子文章）
    📄 人均消费.md     →     📝 人均消费（子文章）
```
