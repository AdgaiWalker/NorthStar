---

## 核心原则

**每个阶段结束，产品都能用。**

- 按依赖分层，逐层交付
- 每个阶段有可验证的里程碑
- AI 写最小可用版本，跑通后再优化
- 测试跟随每个阶段，不积压到最后
- 每周给真人用一次

## 优先级定义

| 标记 | 含义 |
|------|------|
| **P0** | 阶段核心，不做该阶段无法交付 |
| **P1** | 重要体验，影响用户信任 |
| **P2** | 加分项，有数据/有反馈再做 |

## 分层架构

```
第五层：深度 ──── 桌面端三栏 + 全球站 AI 网关（3 天）
                     │
第四层：关系 ──── 通知 + 个人页 + 信任等级（5 天）
                     │
第三层：写 ────── 短帖 + 长文章(AI对话) + 权限（7 天）
                     │
第二层：读 ────── 搜索 + 文章 + 探索 + 空间 + Feed（10 天）
                     │
第一层：地基 ──── 后端骨架 + 路由 + 数据层（5 天）
```

每层严格递进。上层依赖下层。

## 执行原则

### Pieter — 节奏

- 每层结束有里程碑验证
- 每周给真人用一次
- AI 写最小可用版本，跑通后再优化
- 不过度工程——每层只做该层的事

### Adam — 前端

- 不提取组件，除非同一个 UI 模式出现 3 次以上
- AI 写的样式保持内联，不抽 utility function
- 两站样式独立演进
- 桌面端用 `md:` 断点，不单独建页面

### Kent — 质量

- 每层交付带对应测试：API 测试（第一层）、页面测试（第二层）、流程测试（第三层）、触发测试（第四层）
- TypeScript 类型先于实现——prompt 里先附类型定义
- `@ns/shared` 是 API 契约单一来源
- Error boundary + loading 状态随每个页面一起写

| 第七次决策 | 本方案落实 |
