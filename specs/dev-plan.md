## 核心原则

**每个阶段结束，产品都能用。**

- 按依赖分层，逐层交付
- 每个阶段有可验证的里程碑
- AI 写最小可用版本，跑通后再优化
- 测试跟随每个阶段，不积压到最后
- 每周给真人用一次

## 优先级定义

| 标记     | 含义             |
| ------ | -------------- |
| **P0** | 阶段核心，不做该阶段无法交付 |
| **P1** | 重要体验，影响用户信任    |
| **P2** | 加分项，有数据/有反馈再做  |

### 当前全量执行计划（按优先级拆解）

> 目标：在不新增大范围兼容代码、不重做信息架构的前提下，把第一到第四层收口到“可稳定验收”，并把第五层现有雏形完成联调与人工确认。

#### P0：交付阻塞项（必须先完成）

**P0-1 响应式与桌面三栏验收**

- 覆盖页面：首页、搜索页、空间页、文章页、我的页。
- 验收视口：`375px / 768px / 1024px / 1440px`。
- 必修问题：
  - 不允许横向滚动。
  - 不允许按钮区挤压、文字溢出、卡片高度异常跳变。
  - 空间页三栏：左文章树 / 中内容 / 右侧辅助区必须在桌面稳定。
  - 文章页三栏：左同空间文章树 / 中正文 / 右目录与信息区必须稳定。
- 对应任务：`2.16`、`2.17`、`5.1`、`5.2`。

**P0-2 AI 写作闭环补全**

- 维持当前入口不变：仅保留空间页内嵌写作面板，不新增独立写作页。
- 从“主题 -> 草稿 -> 编辑 -> 发布”提升为更清晰的 AI 引导闭环：
  - 第一步：输入主题。
  - 第二步：返回建议方向 / 草稿引导语。
  - 第三步：生成可编辑标题与正文。
  - 第四步：用户确认后发布回当前空间知识区。
- 至少补齐 loading / error / success 三态。
- 对应任务：`3.5`、`3.6`。

**P0-3 第三层测试补强**

- 前端测试补到完整创作闭环：
  - 首页发帖。
  - 空间页发帖。
  - 回复帖子。
  - 求助帖标记“已解决”。
  - AI 生成文章草稿。
  - 发布文章后回到空间知识区可见。
- 对应任务：`3.10`。

**P0-4 通知触发矩阵收口**

- 按现有实现逐项核对并补回归：
  - 内容被确认有帮助。
  - 内容被反馈有变化。
  - 帖子收到回复。
  - 权限 / 信任等级升级。
  - 认证邀请。
  - 过期提醒 / 空间认领。
- 顶栏铃铛与“我的”页通知必须共用同一份状态，已读行为同步。
- 对应任务：`4.2`、`4.9`。

#### P1：稳定性与联调项（P0 后立即执行）

**P1-1 全球站 AI 网关复核**

- 继续保留 `/api/ai/tools` 作为全球站入口。
- 复核 `frontai-web -> /api/ai/tools` 的真实调用链：
  - 请求结构是否与当前后端一致。
  - `tool_choice` / fallback / JSON 返回是否满足全球站现有页面。
  - 异常时是否能优雅降级。
- 对应任务：`5.3`。

**P1-2 工程检查补齐**

- 当前测试、构建、TypeScript 已通过，下一步补 `frontlife-web` lint 能力。
- 由于仓库已是 ESLint 9，优先补 flat config，不走降版本兼容路线。
- 验收标准：
  - `frontlife-web lint` 能跑通。
  - 不为旧 `.eslintrc` 体系写兼容分支。

**P1-3 文档同步**

- 每完成一组任务，回写本节状态面板与勾选清单。
- 保持“计划文档状态”和“实际代码状态”同步，不再出现大段功能已落地、文档仍显示未开始的情况。

#### P2：打磨项（不阻塞交付）

**P2-1 真人试用回合**

- 至少跑一轮真实使用路径：
  - 登录 -> 发帖 -> 回复 -> 解决。
  - 搜索 -> 阅读文章 -> 反馈“有帮助 / 有变化”。
  - AI 写文章 -> 发布。
  - 通知已读。
- 收集具体问题，不做抽象意见记录。

**P2-2 视觉与内容微调**

- 仅针对已发现问题做微调：
  - 文案过长导致布局抖动。
  - 卡片层级不清。
  - 移动端点击热区不足。
  - AI 回答区与本地结果区层次不清。
- 不做新一轮设计重构。

**P2-3 性能与包体观察**

- 当前 `frontlife-web` 构建已通过，但包体偏大，作为后续观察项。
- 若后续真实使用出现首屏慢或切页迟滞，再单独开性能收口任务，不在本轮提前重构。

#### 执行顺序（固定）

1. 先做 `P0-1` 响应式与桌面三栏人工验收并修复。
2. 再做 `P0-2` AI 写作闭环补全。
3. 随后完成 `P0-3` 第三层测试补强。
4. 然后完成 `P0-4` 通知触发矩阵与第四层回归。
5. P0 清空后，执行 `P1-1` 全球站 AI 网关联调。
6. 接着执行 `P1-2` lint 工程检查补齐。
7. 最后做 `P1-3 / P2` 文档同步、真人试用与微调。

#### 完成定义

- P0 全部完成后，校园站可视为“第一到第四层可验收完成”。
- P1 全部完成后，校园站与全球站接口边界可视为“联调完成”。
- P2 只做打磨，不影响是否宣布阶段完成。

## 执行原则

### Pieter — 节奏

- 每层结束有里程碑验证
- 每周给真人用一次
- AI 写最小可用版本，跑通后再优化
- 不过度工程——每层只做该层的事

### Adam — 前端

- 不提取组件，除非同一个 UI 模式出现 3 次以上
- AI 写的样式保持内联，不抽 utility function
- 两站样式独立演进
- 桌面端用 `md:` 断点，不单独建页面

### Kent — 质量

- 每层交付带对应测试：API 测试（第一层）、页面测试（第二层）、流程测试（第三层）、触发测试（第四层）
- TypeScript 类型先于实现——prompt 里先附类型定义
- `@ns/shared` 是 API 契约单一来源
- Error boundary + loading 状态随每个页面一起写
