> 日期：2026-04-22
> 议题：AI 辅助开发模式下的研发方案
> 主导：Pieter Levels
> 与会：Adam Wathan、Kent C. Dodds
> 主持人：AdgaiWalker
> 前置：第七次会议已确定架构（Evan You 主导）

## 一、开场

AdgaiWalker：架构已经定了（Evan You 搞定），技术栈已经定了（React + Tailwind + TypeScript + Hono + Drizzle + PostgreSQL），开发模式是 AI 辅助（Claude Code）。现在需要一个研发方案——先做什么、后做什么、每一步做到什么程度。

---

## 二、各自观点

### Pieter Levels

**核心判断：AI 辅助开发的瓶颈不在写代码，在决定写什么。**

AI 辅助开发最大的坑是过度工程。给 Claude 一个模糊的指令，它会给你 500 行完美架构的代码——但你只需要 50 行。第一条规则：永远先写最小的可用版本，跑通之后再让 AI 帮你优化。

Evan 说的"后端先行"同意，但执行方式要变。传统方式是建数据库→写迁移→写路由→写测试→接前端。AI 辅助方式是先写一个能跑的 JSON 文件模拟 API 响应，前端接上，确认交互没问题，然后一次性让 Claude 把 mock 替换成真实后端。

**核心主张：** 不是 Phase 1/2/3 的线性计划，是一个循环——确定一个用户行为 → 用 AI 写最小实现 → 跑通 → 给真人用 → 看数据 → 下一个行为。

---

### Adam Wathan

**核心判断：逐个替换，不要一刀切。**

校园站数据全在 mock.ts 里。最务实的迁移方式是逐个替换：先把空间列表换成 API 调用，跑通确认没问题，再替换文章详情，再替换帖子。每次替换独立，出问题能回退到 mock。比一次性迁移安全得多。

AI 辅助开发中 Tailwind 的用法要注意：Claude 很擅长写 Tailwind，但倾向于给每个元素加一堆 class。频繁出现的组合抽成组件，出现一两次的保持内联。别让 AI 过早提取组件——那和过早抽象是一样的。

两站 Tailwind 配置差异大（校园站有自定义 token，全球站几乎是空的）。先让校园站跑起来，全球站后面再说。

**核心主张：** 两站样式独立演进，不强求统一。不提取组件，除非同一个 UI 模式出现 3 次以上。

---

### Kent C. Dodds

**核心判断：AI 帮你写代码，但你要能读得懂、改得动。**

AI 辅助开发最大的风险是不可维护。Claude 写的代码你自己不完全理解。三个月后要改一个功能，发现当时 AI 写了一个你没想到的抽象层，改一处牵动全局。没有测试不敢改，没有类型覆盖不知道改错了。

三个保障：

1. **TypeScript 类型先于实现。** 先定义好类型接口，让 AI 按接口实现。看类型签名就知道模块做了什么。
2. **核心路径有测试。** 只测核心循环：搜索→返回结果→打开文章→点"有帮助"。用 Playwright 或 Vitest，写一个端到端测试。
3. **Store 按领域拆分。** 校园站现在一个 Zustand store 管所有状态，功能多了会变成上帝对象。按领域拆：`useSearchStore`、`useSpaceStore`、`useUserStore`。

**核心主张：** "2 周上线"应该是"2 周内有一个核心路径可用的 demo"，不是"2 周内上线生产环境"。没有测试的代码上生产，第一个用户遇到 bug 就再也不会回来。

## 三、讨论

**Pieter：** Kent，我做了 10 年独立开发，上线了十几个产品，没有一个有自动化测试。测试在用户验证之后，不在之前。

**Kent：** 你做的是工具型产品（Nomad List 是搜索+列表），用户遇到 bug 刷新就好。盘根的核心是信任——"确认有帮助"这个数据如果出错，用户信任就崩了。内容平台和工具不一样。

**Adam：** 你们两个都对，只是阶段不同。Pieter 说的是"做出来"的阶段，Kent 说的是"扛得住"的阶段。关键问题是：校园站"一个食堂" MVP，到底需要扛得住什么？如果是 10 个同学在食堂门口扫码试用——mock 够了。如果是上线到全校——需要测试。

**Pieter：** 不管瞄哪个，第一步是一样的——先让一个人用起来。找一个同学，让他搜一个问题。看他能不能搜到、能不能看文章、会不会点"有帮助"。这一步连后端都不需要，mock 就够了。

**Kent：** 同意第一步是让一个人用。但我加一个条件：把这个人用了什么、卡在哪里、有没有点"有帮助"全部记录下来。不是凭感觉，是看数据。

**Adam：** 我来梳理一下，按 AI 辅助开发的节奏——

第一周：核心路径跑通（mock 数据）
第二周：接真实后端
第三周：给更多人用，记录行为数据

**Pieter：** 第一周不应该做搜索结果三种模式（精确匹配/无匹配/部分匹配）。那是 PRD 的完整版。第一周只做一种：搜到就显示文章，搜不到就显示"暂时没有"。砍掉 AI 回答和部分匹配。

**Kent：** 同意砍。而且第一周甚至不开 AI 回答。搜不到就显示"暂时没有内容，你可以来补充"。校园站第一周只有 3-5 个空间、20-30 篇文章。搜索只在这些内容里匹配就够了。PRD 说"数据诚实原则"——没有足够内容时 AI 回答质量不好，不如先诚实地说"没有"，等种子内容到位了再开 AI。

**Adam：** 这个建议很关键。第一周先把"搜到→看→确认"这条纯本地路径跑通。AI 回答晚一周开。

## 四、最终方案

**原则：一个人用一个功能，比一百个人看一个 demo 有价值。**

### 第一周：一个人搜到一篇文章并点了"有帮助"

**核心路径：**
```
学生打开 → 搜"食堂" → 看到"二食堂麻辣烫测评" → 点进去 → 读完 → 点"有帮助"
```

**只做：**
- 首页搜索框
- 搜索结果（仅本地匹配，无 AI）
- 文章阅读页 + "有帮助"按钮
- mock 数据，mock 3-5 篇食堂相关文章

**不做：** 推荐卡片、信息流、帖子、AI 回答、后端、空间页、个人页、桌面端

**验证：** 找一个同学，让他搜"食堂"，看他能不能走完全程。

---

### 第二周：接后端 + 开 AI 回答

**做：**
- Claude 搭 Hono 后端，5 个端点（见第七次会议）
- 逐个替换 mock（用 `VITE_USE_MOCK` 切换）
- 开启 AI 搜索兜底（搜不到时 AI 回答）
- 种子内容扩展到 20 篇

**替换顺序：** 空间列表 → 文章详情 → 帖子。每次替换独立，出问题回退到 mock。

**验证：** 10 个同学试用，记录搜索 query 和点击行为。

---

### 第三周：根据数据迭代

**做：**
- 看第二周数据：哪些 query 搜不到？哪些文章没人看？
- 补内容、调搜索、修交互
- 加推荐卡片（基于真实点击数据，不是编辑精选）

**验证：** 有没有人第二次回来用。

---

### 执行原则

**Pieter 的节奏原则：**
- 每周一个可验证的目标
- 每周给真人用一次
- 数据驱动下一步，不靠猜
- AI 写最小可用版本，跑通后再优化

**Adam 的技术原则：**
- 不提取组件，除非同一个 UI 模式出现 3 次以上
- AI 写的样式保持内联，不抽 utility function
- 两站样式独立演进，不强求统一
- 逐个替换 mock，不要一刀切

**Kent 的质量原则：**
- 核心路径写端到端测试（搜索→文章→确认），CI 跑着
- TypeScript 类型先于实现——先定义 API 响应类型，再让 Claude 写路由
- Store 按领域拆分，不让一个 store 管所有状态
