# 第七次产品委员会会议纪要

> 日期：2026-04-22
> 议题：盘根双产品架构设计
> 与会：Evan You（尤雨溪）、Guillermo Rauch、Jeff Atwood
> 主持人：AdgaiWalker
> 决策权：Evan You（AdgaiWalker 指定主导）

---

## 一、开场

AdgaiWalker：盘根有两个产品——校园站和全球站。需要一个架构师来决定整体技术架构。今天请三位来审阅代码库，给出各自的判断，然后讨论出结论。

---

## 二、各自观点

### Evan You（尤雨溪）

**核心观察：**

1. **Monorepo 结构松散但合理。** 没有 turbo/nx，每个包独立 build。原型阶段没问题，但 `@ns/shared` 直接 export TS 源码（没有 build 步骤），上线前需要处理。

2. **两站架构差距大。** 校园站一个 store、无 React Query、无表单库——轻。全球站三个 store、React Query、React Hook Form + Zod、CodeMirror、dnd-kit——重。这个差距是合理的，但 `@ns/shared` 目前只有类型和常量，没有承担起"共享逻辑层"的职责。

3. **代码没有体现"渐进式"。** PRD 里一切都是渐进的——信任等级渐进、内容渐进、AI 从无感到有感。但代码上，全球站已经建了完整的审核引擎（WIP limit、SLA 跟踪、自动分配），校园站还全是 mock 数据。这不是渐进，是跳步。

**核心主张：** 从骨架开始，但骨架要能长肉。后端先建路由，共享包先建契约，前端先砍掉复杂度。全球站的审核引擎等校园站需要时再共享。过早抽象比没有抽象更危险。

---

### Guillermo Rauch

**核心观察：**

1. **部署架构需要现在就想清楚。** 两个前端独立部署，后端通过 `SITE=cn|com` 切换——单实例多租户。两个后果：数据库必须隔离（物理还是逻辑待定），共享包变更需要自动化类型检查。

2. **AI 层应该在后端统一。** 两站都 proxy `/__ai` 到智谱，但方式不同（校园站 SSE 流式，全球站 function calling）。差异是合理的，但前端不应该知道 AI 服务的细节。后端做网关，前端做薄客户端。

3. **最大的风险：后端只有 schema 没有路由。** 10 张表定义了但没有 CRUD。两站前端都在 mock 积债。越往后拖，迁移成本越高。

**核心主张：** 后端优先。先把 5 个 API 端点跑通，让前端至少有一条真实的 CRUD 路径。AI 网关统一到后端，前端不持有模型 ID、不组装 prompt。

---

### Jeff Atwood

**核心观察：**

1. **信任等级太复杂。** Schema 有 6 级（guest/user/active/author/senior/admin），还有 `auth_requests` 表做认证流程。Discourse 的 Trust Level 系统有 5 级、每级十几个权限开关，花了两年才调对。建议先砍到 3 个：访客、已登录、管理员。Author 和 senior 是运营出来的，不是设计出来的。

2. **内容层级要限死。** 空间包含文章、文章可以嵌套（parentId 自引用），这个结构对。但深度最多两层。Discourse 的 category → subcategory 也是两层，够了。帖子扁平归属空间。

3. **搜索日志太浅。** `search_logs` 只有四个字段（query, 结果数, 是否AI回答, 时间）。搜索是产品的心脏，日志应该记录更多：点击了哪个结果、停留多久、是否点了"有帮助"。

**核心主张：** 简化信任等级，加深搜索日志，内容层级限两层。

---

## 三、讨论

**Evan：** Jeff 说得对，6 个信任等级太复杂。前端只需要知道"能不能做某件事"，不需要知道"你是第几级"。后端加一个 `GET /api/me/permissions`，返回权限列表。前端不关心等级数字。

**Guillermo：** 同意。后端管等级，前端查权限。

**Jeff：** Discourse 后来也是这么改的。Trust Level 是后端的事，前端只有"显示/隐藏"。

**Evan：** 关于共享包，边界是合理的——纯 TS、纯类型和常量。不往里加业务逻辑。两站业务差异大，强行共享变成耦合。如果未来有真正的共享逻辑（敏感词过滤、搜索日志），可以加，但要有明确边界。

**Guillermo：** AI 网关应该在后端：接收前端请求 → 根据 SITE 选系统提示词 → 加上下文 → 调智谱 → 返回。前端 AI 服务变成薄 HTTP 客户端。

**Jeff：** 搜索日志建议加 `search_events` 表，记录 query、点击结果、停留时长、是否确认。冷启动期数据是空的，但架构先留好。到有数据时，"编辑精选 → AI 推荐"的切换顺理成章。

**Evan：** 我总结三个共识、三个分歧——

**共识：**
1. 后端先行——先建路由和 API
2. 信任等级简化——后端管等级，前端只查权限
3. AI 网关统一——后端代理 AI，前端只管展示

**分歧：**
1. 搜索日志深度（Jeff 说现在建全量表，Evan 说等有数据再扩展）
2. 审核引擎是否拆到共享包（Guillermo 说拆，Evan 说不拆）
3. 数据库物理隔离 vs 逻辑隔离（Guillermo 说物理，Evan 说原型期逻辑够了）

## 四、Evan You 的裁决

AdgaiWalker 指定 Evan You 主导所有决策。裁决如下：

### 裁决 1：搜索日志——先用简单日志，预留扩展口

Schema 保留现有 `search_logs` 四字段。后端路由层留好 hook，等真实搜索流量上来（DAU > 100）时无缝扩展 `search_events` 表。不提前建表，但提前设计好接口。

**理由：** 冷启动期搜索量接近零，建了全量表也是空的。但接口设计要从一开始考虑扩展性——路由函数的参数和返回值要能容纳未来的字段。

### 裁决 2：审核引擎——留在全球站内部

审核逻辑留 frontai-web，不拆到 shared。

**理由：** 校园站现阶段不需要审核（先发后审 + 团队人工检查）。过早抽象带来的维护成本高于收益。等校园站用户量增长到需要自动化审核时，再从全球站提取共享模块。到那时需求更明确，抽象更精准。

### 裁决 3：数据库——原型期逻辑隔离，上线前物理隔离

开发阶段用同一个 PostgreSQL 实例，`site` 字段区分数据。上线前迁移到独立实例。

**理由：** 渐进式架构。现在不花运维成本，但 schema 设计从一开始就不依赖"在同一个库里"——所有查询都带 `site` 条件，迁移时只需要改连接字符串。

## 五、实施计划

### Phase 1：后端骨架

新建 Hono 入口 + 5 个 API 端点 + Drizzle 迁移配置。

| 端点 | 用途 |
|------|------|
| `GET /api/spaces` | 空间列表 |
| `GET /api/spaces/:id` | 空间详情 + 文章列表 |
| `GET /api/articles/:id` | 文章详情 |
| `POST /api/posts` | 发帖 |
| `GET /api/me/permissions` | 当前用户权限 |

### Phase 2：校园站对接后端

- AIService 改为调 `/api/ai/search`
- 新增 `services/api.ts` 薄 HTTP 客户端
- useAppStore 移除 mock 依赖，mock 与 API 通过环境变量共存（`VITE_USE_MOCK=true`）
- Vite config 移除 AI proxy，改为 proxy 到后端

### Phase 3：权限系统

- 后端实现 `GET /api/me/permissions`
- 前端 store 加 permissions 状态，替代硬编码的 isLoggedIn/isCertified

## 七、三人审理修正

### 修正 1：Mock 与 API 共存（Evan 提出）

Phase 2 "移除 mock 依赖"表述不准确。冷启动期需要 mock 数据开发。正确做法：mock 和 API 共存，通过环境变量切换（`VITE_USE_MOCK=true`），不是一刀切。

### 修正 2：AI 网关分两个端点（Guillermo 提出）

两站 AI 调用方式不同，不是一个统一接口能搞定：
- `POST /api/ai/search` — 校园站搜索（SSE 流式返回）
- `POST /api/ai/tools` — 全球站工具推荐（JSON 返回 + function calling）

会议纪要原文"后端根据 SITE 选系统提示词"过于简化，需修正。

### 修正 3：PRD 补充内容（Jeff 提出）

- 信任等级：PRD Section 26 保留 6 级表格，但需标注"初期活跃等级"为 3 个（Visitor / Registered / Admin），其余等级等运营需要时激活
- 内容层级：空间页补充"文章嵌套深度最多两层"
