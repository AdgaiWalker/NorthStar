## 0. 已完成清单（不在本计划范围）

以下能力已实现并通过类型检查，不再展开：

- **Identity**：注册、登录、me、GitHub OAuth、密码重置、邮箱验证、邀请码、申请制
- **Capability**：`GET /api/platform/capabilities?site=campus|compass` 两站能力查询
- **AI Gateway**：敏感词双检、额度检查、6 种 fallback、token 日志、10% AI 输出抽检
- **Campus 完整闭环**：空间、文章（嵌套）、帖子、回复、有帮助、有变化、搜索三态、Feed、收藏、AI 写作、通知、法律页、数据导出、注销
- **Compass 完整闭环**：内容 CRUD + 版本、方案（目标优先）+ 保存 + 导出 + 反馈、搜索、收藏、用户中心 6 tab
- **Moderation**：任务 CRUD、5 态状态机、审核副作用
- **Notification**：站内通知 + 邮件 provider（dev）+ 投递记录
- **Analytics**：18 种行为事件、指标聚合、90 天清理脚本
- **Billing**：额度、账本、手动支付
- **Compliance**：法律文档、同意、数据导出、账号注销（含擦除）
- **Admin 11 页**：总览、审核、用户、内容、审计、配置、数据、支付、投递
- **校园站 mock 清除**：全部 API 真实化
- **通知 6 种类型全覆盖**：feedback（`postgres.ts:529`）、reply（`postgres.ts:460`）、changed（`moderation/service.ts:82`）、claim（`campus/repository.ts:187`）、expiry（`postgres.ts`）、auth_invite（`postgres.ts`）
- **全球站 AI 调用已走 Gateway**：`AIService.ts` 通过 `API_ENDPOINTS.AI_CHAT` = `/api/ai-gateway/chat` 调用后端，不直连智谱

## 2. R1：功能闭环补全

> 补齐差异分析中标记为 ❌ 的前端缺失项。这些是文档要求但代码中完全没有的页面或流程。

**差距**：`/legal/terms` 和 `/legal/privacy` 有导航按钮但无路由、无页面。

**落点**：
- `frontai-web/src/pages/LegalPage.tsx` — 新建

- [ ] 点击设置页"用户协议"/"隐私政策"可到达对应页面
- [ ] 页面展示文档内容、版本号、发布日期
- [ ] Footer 中法律链接可达

### R1-2 全球站密码重置入口

**差距**：后端 `/api/identity/password-reset/*` 已实现，前端无入口。

## 4. R3：平台治理

> 生产级基础设施收口。

### R3-1 生产级邮件 Provider

**差距**：当前仅 `devEmailProvider`（内存日志），无真实邮件发送。

**落点**：
- `server/src/modules/notification/email-provider.ts` — 新增 `smtpEmailProvider`
- 通过环境变量配置 SMTP（host、port、user、pass、from）
- 保留 `devEmailProvider` 作为无配置时的 fallback

**验收**：
- [ ] 配置 SMTP 后邮件可真实发送
- [ ] 无配置时降级到 dev provider，不白屏
- [ ] 发送失败有重试和记录

### R3-2 行为数据 90 天自动清理

**差距**：`cleanup-analytics.ts` 脚本存在但需手动执行。

**落点**：
- `server/src/index.ts` — 启动时注册定时任务（每日凌晨执行清理）
- 或通过外部 cron 调用 `pnpm cleanup:analytics`

**验收**：
- [ ] 90 天以上行为事件自动清理
- [ ] 清理动作写审计日志

### R3-3 Moderation 任务去重

**差距**：无去重逻辑，重复 pending task 可积累。

**落点**：
- `server/src/modules/moderation/repository.ts` — `createModerationTask` 前检查同 `site + type + targetType + targetId + status=pending` 是否已存在
- 存在时跳过创建，返回已有记录

**验收**：
- [ ] 同一目标不会产生重复 pending task

### R3-4 旧路由清理

**差距**：`src/routes/auth.ts`（`/api/auth/*`）为旧路径。

**落点**：
- 确认前端已全部切换到 `/api/identity/*`
- 删除 `src/routes/auth.ts`
- 从 `index.ts` 移除挂载
- 全仓搜索确认无依赖

**验收**：
- [ ] `/api/auth/*` 不再存在
- [ ] 两站登录/注册不受影响

## 6. R5：技术债

> 延后处理的结构性工作。

### R5-1 Identity 模型拆分

**差距**：架构设计要求 Account + Credential + CampusProfile + CompassProfile + LevelChangeLog 五个模型，当前为单 `users` 表。

**评估**：功能不受影响，当前阶段无实际风险。仅在需要多绑定方式或多 profile 差异化时才有必要。

**建议**：P7 执行，或在实际需求出现时再做。

### R5-2 学校报告工作台

**差距**：架构设计提到但无具体 PRD 细节。

**建议**：待校园站运营后有实际数据需求时再设计。

### R5-3 支付 Provider 接入

**差距**：当前仅手动确认。

**建议**：上线后按需接入 Stripe/微信/支付宝。

### R5-4 Moderation 驳回原因分离

**差距**：仅有单一 `reason` 字段，架构要求内部原因 + 可展示原因分离。

**落点**：
- `moderationTasks` 表新增 `internalReason` 字段
- `reason` 作为用户可见原因
- 后台审核时可分别填写
