# 盘根 · AI 指南针（PanGen AI Compass）项目宪法

## 核心原则

### 一、技术栈不变量

前端技术栈锁定如下，偏离须经书面论证并通过修订流程审批。

- 语言：TypeScript
- 框架：React（当前 React 19）
- 构建：Vite（支持多入口：校园站 + 全球站）
- 路由：React Router（当前 v7）
- 样式：Tailwind CSS
- 全局状态：Zustand（轻量、无仪式感）
- 表单：React Hook Form + Zod 校验
- Markdown 渲染：react-markdown + remark-gfm + rehype-raw / rehype-sanitize
- UI 组件：shadcn/ui（允许渐进引入，不强制一次性改造）
- AI 模型：智谱 GLM-4-Flash（通过后端 AI 网关代理调用）

**不写兼容代码**：如无显性兼容性要求，禁止编写额外兼容代码（如旧浏览器
polyfill、降级分支）。优先使用现代浏览器能力，以桌面 Chrome、移动端
Safari / Chrome 为主。

**后端演进**：前端原型阶段允许使用 localStorage + Mock 数据闭环体验。
待需要多人协作或成本治理时再引入后端服务与数据库——原型阶段不以后端为阻塞。
后端前期使用 Node.js + Hono.js + Prisma + PostgreSQL 快速验证业务，
后期按模块逐步迁移至 Golang + Gin + sqlc（API 契约保持不变，前端无感）。

### 二、安全与密钥不变量

- 前端构建产物不得包含任何第三方模型厂商的真实 API Key（明文或混淆均禁止）。
- 开发环境：API 请求仅通过 Vite dev proxy 转发。
- 生产环境：所有 AI 调用必须经后端 AI 网关，由网关持有 Key。
- 含密钥的本地配置文件（`.env.*.local`、`.zhipu.local.json` 等）必须在
  `.gitignore` 中声明，绝不提交到版本控制。

**原因**：API Key 泄露直接造成经济损失，威胁项目运营安全。

**AI 资源治理**：AI 搜索 MUST 从第一天起实施限流，先严后松。
- 未登录用户：每日搜索上限 MUST 低于登录用户。
- 登录用户：有明确额度，额度耗尽给出提示而非静默失败。
- 异常检测：同一 IP 短时间大量请求 MUST 自动拦截。
- 额度标准上线后可调整，但不允许存在"不限量"状态。

**原因**：AI 调用按次计费，无防护直接暴露于公网等于财务风险。

### 三、文档与语言不变量

- 文档（规格、计划、变更记录）尽量使用中文编写。
- 代码注释尽量使用中文编写。
- 变量 / 函数 / 类型命名使用英文驼峰（camelCase / PascalCase）。
- 本宪法以中文为权威文本。

**原因**：团队沟通以中文为主；代码标识符保持英文以兼容生态与工具链。

### 四、产品体验不变量（Apple 式克制）

- **最短主路径**：任何页面最多 1 个主按钮；主按钮表达"下一步行动"。
- **渐进披露**：默认只展示决策所需信息，细节通过抽屉 / 展开呈现。
- **一致性优先**：同类组件在不同页面保持一致（错误态、空态、确认弹窗）。
- **反馈及时**：关键操作必须给出 loading / 成功 / 失败 / 不可撤销提示。
- **禁止**：通过"仪式感文案 / 装饰性动效"制造价值感。
- **冷启动诚实策略**：内容不足时，AI 搜索无结果 MUST NOT 编造内容。
  无结果时必须展示相关已有内容 + 收集用户需求入口。每个领域至少
  有 10 篇内容后才正式开放（开放前使用 Locked 态）。

**原因**：用户群包含非技术用户；清晰可预期减少支持负担、建立信任。

### 五、多端交付不变量

- 桌面端：Web（浏览器）为主形态。
- 移动端 App：独立设计、独立开发，与 Web 完全不同，仅通过 API 共享数据。
- App 技术选型不与 Web 绑定，待 Web 站稳定后单独评估。
- Web 端移动适配底线：触控目标 ≥ 44×44 px；Safe Area 不遮挡；
  键盘弹出不挡主操作。

**原因**：Web 是核心产品（信息网站），App 是配套产品。两者 UI/交互
天然不同，强求统一会两头不讨好。共享 API 数据层即可。

### 六、功能阶段化上线不变量（Feature Flags）

所有非核心能力必须可阶段化点亮，统一三态策略：

1. **Hidden（隐藏）**：入口不可见。
2. **Locked（锁定态）**：入口可见但不可用，点击给出解释与预期上线时间。
3. **Degraded（降级可用）**：功能可用但质量 / 额度 / 范围降低，必须有明确提示。

核心能力（阅读内容、导航、基础搜索）免于 Feature Flag 门控。

**原因**：分阶段上线防止未完成功能污染用户体验，同时提供安全的回滚手段。

### 七、内容与文档标准不变量

- 系统内部唯一标准内容格式：Markdown。
- 允许受控扩展块（白名单）：例如 `:::video{...}:::`、`:::callout{...}:::`。
- 禁止任意 HTML 直出（安全与样式失控）。

**原因**：Markdown 保持内容管道确定性，避免未过滤 HTML 带来的 XSS 风险。

### 八、后台与协作不变量

- 系统后台（`/admin`）是高权限区：仅 reviewer / editor / admin 可访问；
  普通作者不可见。
- 内容发布必须审核通过（可由任意授权审核员完成，不强制本人）。
- 发布后允许修改，但必须走二次审核；线上旧版本保持可见，新版本审核通过后替换。
- 后台 AI 只做"建议 / 草稿 / 辅助分析"，禁止自动发布、自动精选、自动下架。
- 指派 / 改派 / AI 操作 / 版本回滚必须写审计日志。

**原因**：内容完整性与问责机制对面向用户群体的平台不可妥协。

### 九、本地存储不变量

- 读写必须有校验（guard），数据损坏自动重置。
- 重置必须提示一次且文案固定："本地数据异常，系统已重置到默认状态。"
- 写入必须满足原子性：一次 `setItem` 写入完整 JSON，禁止分多次局部更新。

**原因**：原型阶段 localStorage 是当前持久层，保证数据完整性可防止脏数据引发级联 UI 错误。

### 十、参考规格与单一事实来源

- 单一事实来源（PRD）：`specs/PRD-盘根AI指南针-标准版.md`
- 变更规格：`specs/` 下各功能目录。
- 本宪法是所有开发决策的最高约束文件。规格文档与宪法冲突时，以宪法为准。

### 十一、双站架构：一半生活一半理想

项目采用双前端架构，两个站点共享产品能力，仅内容领域和数据合规不同。

#### 核心理念：一半生活，一半理想

- **生活面（xyzidea.cn）**：以校园为锚点的区域知识平台——日常起居 /
  成长提升 / 精明消费。人生无限可能，不止于学生和老师，而是校园所在
  区域的每一个人。
- **理想面（xyzidea.com）**：XYZ 无限可能——AI 前沿知识、工具方法、
  创新实践。你的 idea 可以变成现实，能够创新，能够解决真实问题，
  能够创造实际价值。

两站共享产品能力与交互体验，面向不同场景和用户群体。底层由 Ferry
框架（熵减思维：规划→执行→复盘）指导两站的内容方向与质量保证。

#### 站间链接规则

- 用户可在两站之间自由跳转，新标签页打开
- 跳转不传递登录态（两站用户体系完全独立）
- 跳转后可继续浏览/搜索/收藏（需在目标站注册登录）

#### 数据隔离红线（最高优先级）

以下原则**绝对不能违背**：

1. **cn 用户数据绝不流向海外节点** — 手机号等个人信息只能存于国内 PostgreSQL
2. **com 站点禁止收集任何中国敏感个人信息** — 绝对禁止：手机号、微信号、QQ号、身份证、生物特征
3. **用户账号体系完全独立** — 两站用户互不相通，不做单点登录
4. **账号注销后 30 天内物理删除** — CDC 确保两边同步清理
5. **内容同步为单向只读** — cn 站写，com 站只读同步，不反向写入

#### 内容本地化策略

- 创作语言统一为中文（创作者母语，产出效率最高）
- 内容发布时 MUST 经过 AI 翻译（中文 → 英文），提供双语版本
- 翻译后的文章状态为"待审校"，仅标题和摘要需人工审校
- 翻译使用智谱 GLM-4-Flash 完成，不引入额外翻译服务

**原因**：双站架构兼顾国内合规和全球服务。校园站以区域化知识服务生活（人生无限可能），全球站以 AI 知识服务创新（Idea 无限可能）。两站共享产品能力降低维护成本，内容差异化确保精准服务各自用户群体。

### 十二、数据平台不变量

数据平台 MUST 按角色分离视图，每个角色只看到与决策相关的数据：

#### 三角色定义

- **研究员**（内容方向决策）：关注搜索查询分析、内容覆盖率、
  阅读行为、用户来源去向、趋势变化。核心问题："下一篇该写什么？"
- **运营**（内容上线与增长）：关注内容管道状态、内容表现排名、
  用户反馈、Feature Flag 状态。核心问题："内容健康吗？"
- **开发**（系统稳定）：关注 API 性能与错误率、AI 网关用量与成本、
  部署状态。核心问题："系统正常吗？"

#### 数据采集原则

- 采集 MUST 以后端为主、前端为辅，避免前端埋点遗漏。
- 采集的数据 MUST 可追溯至具体用户行为（需用户登录）。
- 匿名流量只记录聚合指标（PV、搜索量），不记录个人行为。

#### 用户留存核心机制

- Topic 关注：用户关注感兴趣的话题，新文章发布时通知关注者。
- 这是唯一优先实现的留存机制，其他留存手段（搜索历史、个性化推荐）
  待此机制验证有效后再叠加。

**原因**：数据驱动决策需要结构化的数据采集和角色化的数据视图。
Topic 关注与三级内容结构（Domain → Topic → Article）天然契合，
实现成本低且可直接验证留存效果。

## 开发工作流

1. **规格先行**：每个功能先在 `specs/` 下形成图纸，
   再进入代码实现。无规格不动代码。
2. **宪法关卡**：实现前必须逐条核对上述原则是否合规；违规项必须在计划的
   复杂度追踪表中显式标注。
3. **双站验证**：涉及两站的功能变更 MUST 同时验证校园站和全球站，
   确保功能对称性和交互一致性。
4. **渐进交付**：按优先级实现用户故事（P1 → P2 → P3），每个故事必须可独立
   测试、独立部署。
5. **缓存清理**：修改 Vite 配置或安装新依赖后，必须删除
   `node_modules/.vite` 目录再重启 dev server，避免 Outdated Optimize Dep
   错误。
6. **发布审核**：内容变更需审核员通过；技术变更需宪法合规检查。
7. **审计留痕**：所有后台操作、AI 建议、内容版本变更必须记录日志。

## 治理

本宪法高于所有其他开发实践、临时讨论与个人偏好。

**修订流程**：
- 以规格文档形式在 `specs/` 下提出修订提案。
- 提案必须包含：修订理由、受影响原则、现有代码迁移方案、审批签字。
- MAJOR 版本：原则删除或重新定义（向后不兼容）。
- MINOR 版本：新增原则或实质性扩展。
- PATCH 版本：措辞澄清、排版、笔误修正。

**合规审查**：每次 PR 或评审必须验证宪法合规性。未经论证的复杂度违规必须在合入前解决。

**运行时指引**：日常开发使用 `.specify/` 模板与命令。模板与宪法冲突时，以宪法为准。
