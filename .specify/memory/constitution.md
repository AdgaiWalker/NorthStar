# 盘根 · AI 指南针（PanGen AI Compass）项目宪法

## 核心原则

### 一、技术栈不变量

前端技术栈锁定如下，偏离须经书面论证并通过修订流程审批。

- 语言：TypeScript
- 框架：React（当前 React 19）
- 构建：Vite（双入口：`vite.config.ts` 全球站前端 +
  `vite.config.campus.ts` 校园站前端）
- 路由：React Router（当前 v7）
- 样式：Tailwind CSS
- 全局状态：Zustand（轻量、无仪式感）
- 表单：React Hook Form + Zod 校验
- Markdown 渲染：react-markdown + remark-gfm + rehype-raw / rehype-sanitize
- UI 组件：shadcn/ui（允许渐进引入，不强制一次性改造）
- AI 模型：智谱 GLM-4-Flash（两站共用，通过后端 AI 网关代理调用）

**pnpm 严格模式 React 去重**：pnpm 严格模式下 `react-router-dom`
等 peer dependency 会在自身 `node_modules` 内嵌 React 副本，
导致 `Invalid hook call`。校园前端 `vite.config.campus.ts` 必须
通过 `resolve.alias` 显式将 `react`、`react-dom`、
`react/jsx-runtime` 指向同一份实例。全球站前端通过 `dedupe`
配置处理。新增依赖引入 React peer dependency 时必须验证两个
前端均无 React 重复实例。

**不写兼容代码**：如无显性兼容性要求，禁止编写额外兼容代码（如旧浏览器
polyfill、降级分支）。优先使用现代浏览器能力，以桌面 Chrome、移动端
Safari / Chrome 为主。

**后端演进**：前端原型阶段允许使用 localStorage + Mock 数据闭环体验。
待需要多人协作或成本治理时再引入后端服务与数据库——原型阶段不以后端为阻塞。
后端前期使用 Node.js + Hono.js + Prisma + PostgreSQL 快速验证业务，
后期按模块逐步迁移至 Golang + Gin + sqlc（API 契约保持不变，前端无感），
详见原则十一。

### 二、安全与密钥不变量

- 前端构建产物不得包含任何第三方模型厂商的真实 API Key（明文或混淆均禁止）。
- 开发环境：API 请求仅通过 Vite dev proxy 转发。
- 生产环境：所有 AI 调用必须经后端 AI 网关，由网关持有 Key。
- 含密钥的本地配置文件（`.env.*.local`、`.zhipu.local.json` 等）必须在
  `.gitignore` 中声明，绝不提交到版本控制。

**原因**：API Key 泄露直接造成经济损失，威胁项目运营安全。

### 三、文档与语言不变量

- 文档（规格、计划、变更记录）尽量使用中文编写。
- 代码注释尽量使用中文编写。
- 变量 / 函数 / 类型命名使用英文驼峰（camelCase / PascalCase）。
- 本宪法以中文为权威文本。

**原因**：团队沟通以中文为主；代码标识符保持英文以兼容生态与工具链。

### 四、产品体验不变量（Apple 式克制）

- **最短主路径**：任何页面最多 1 个主按钮；主按钮表达"下一步行动"。
- **渐进披露**：默认只展示决策所需信息，细节通过抽屉 / 展开呈现。
- **一致性优先**：同类组件在不同页面保持一致（错误态、空态、确认弹窗）。
- **反馈及时**：关键操作必须给出 loading / 成功 / 失败 / 不可撤销提示。
- **禁止**：通过"仪式感文案 / 装饰性动效"制造价值感。

**原因**：用户群包含非技术学生与教师；清晰可预期减少支持负担、建立信任。

### 五、多端交付不变量

- 桌面端：Web（浏览器）为主形态。
- 移动端 App：独立设计、独立开发，与 Web 完全不同，仅通过 API 共享数据。
- App 技术选型不与 Web 绑定，待 Web 站稳定后单独评估。
- Web 端移动适配底线：触控目标 ≥ 44×44 px；Safe Area 不遮挡；
  键盘弹出不挡主操作。

**原因**：Web 是核心产品（信息网站），App 是配套产品。两者 UI/交互
天然不同，强求统一会两头不讨好。共享 API 数据层即可。

### 六、功能阶段化上线不变量（Feature Flags）

所有非核心能力必须可阶段化点亮，统一三态策略：

1. **Hidden（隐藏）**：入口不可见。
2. **Locked（锁定态）**：入口可见但不可用，点击给出解释与预期上线时间。
3. **Degraded（降级可用）**：功能可用但质量 / 额度 / 范围降低，必须有明确提示。

核心能力（阅读内容、导航、基础搜索）免于 Feature Flag 门控。

**原因**：分阶段上线防止未完成功能污染用户体验，同时提供安全的回滚手段。

### 七、内容与文档标准不变量

- 系统内部唯一标准内容格式：Markdown。
- 允许受控扩展块（白名单）：例如 `:::video{...}:::`、`:::callout{...}:::`。
- 禁止任意 HTML 直出（安全与样式失控）。

**原因**：Markdown 保持内容管道确定性，避免未过滤 HTML 带来的 XSS 风险。

### 八、后台与协作不变量

- 系统后台（`/admin`）是高权限区：仅 reviewer / editor / admin 可访问；
  普通作者不可见。
- 内容发布必须审核通过（可由任意授权审核员完成，不强制本人）。
- 发布后允许修改，但必须走二次审核；线上旧版本保持可见，新版本审核通过后替换。
- 后台 AI 只做"建议 / 草稿 / 辅助分析"，禁止自动发布、自动精选、自动下架。
- 指派 / 改派 / AI 操作 / 版本回滚必须写审计日志。

**原因**：内容完整性与问责机制对面向师生群体的教育平台不可妥协。

### 九、本地存储不变量

- 读写必须有校验（guard），数据损坏自动重置。
- 重置必须提示一次且文案固定："本地数据异常，系统已重置到默认状态。"
- 写入必须满足原子性：一次 `setItem` 写入完整 JSON，禁止分多次局部更新。

**原因**：原型阶段 localStorage 是当前持久层，保证数据完整性可防止脏数据引发级联 UI 错误。

### 十、参考规格与单一事实来源

- 单一事实来源（PRD）：`specs/PRD-盘根AI指南针-标准版.md`
- 变更规格：`NorthStar/openspec/Changes/` 下最新变更目录。
- 本宪法是所有开发决策的最高约束文件。规格文档与宪法冲突时，以宪法为准。

### 十一、双面架构不变量

项目以同一代码仓库提供两个独立前端，面向不同用户群和合规要求：

#### 功能一致性原则

两站功能完全一致，仅内容不同。共享所有产品能力（布局、搜索、AI 搜索、
方案生成、收藏、导出等），区别仅在于内容领域和数据合规要求。

#### 站点定义

- **校园生活面**（Campus Face）：
  - 域名：`xyzidea.cn`（国内服务器，ICP 备案）
  - 入口：`campus.html` + `vite.config.campus.ts`
  - 路由前缀：`/`（独立 BrowserRouter）
  - 目标用户：在校大学生
  - 内容领域：`daily`（日常起居）、`growth`（成长提升）、`deal`（精明消费）
  - 内容组织：Domain → Topic（专题） → Article（三级结构，与全球站一致）
  - AI 搜索：智谱 GLM-4-Flash，系统提示定位为"校园生活顾问"
  - 合规要求：国内合规提供商（智谱 AI）；禁止境外服务依赖

- **全球 AI 工具面**（Global AI Face）：
  - 域名：`xyzidea.com`（海外服务器）
  - 入口：`index.html` + `vite.config.ts`
  - 路由前缀：`/`（独立 BrowserRouter）
  - 目标用户：全球想用 AI 解决问题的人
  - 核心理念：靠 AI 解决问题，不是单纯学 AI
  - AI 搜索：智谱 GLM-4-Flash，系统提示定位为"AI 软件顾问"
  - 内容领域：`creative`（影视创作）、`dev`（编程开发）、`work`（通用办公）

#### 功能对称表

| 功能 | 全球站 | 校园站 |
|------|--------|--------|
| 首页布局 | Hero + AI 搜索 + 领域切换器 + 模块切换器 | 完全一致 |
| AI 智能搜索 | 智谱 GLM-4-Flash | 完全一致（不同系统提示） |
| 普通搜索 | 关键词过滤 | 完全一致 |
| 领域切换器 | 影视创作/编程开发/通用办公 | 日常起居/成长提升/精明消费 |
| 模块切换器 | 热门推荐/专题系列 | 完全一致 |
| 文章页三栏布局 | 系列目录 + 正文 + TOC | 完全一致 |
| 方案生成/导出 | 支持 | 支持 |
| 站间链接 | Header 链接到校园站 | Header 链接到全球站 |

#### 隔离规则

- 两个前端 MUST NOT 共享导航组件（Header、Footer、BottomNav 各自独立）。
- 共享代码限限于无业务逻辑的工具层：`src/components/DocRenderer`、
  `src/utils/`、`src/types/`。
- 共享代码的修改 MUST 同时验证两个前端不受破坏。
- 后端前期使用 Node.js + Hono.js + Prisma + PostgreSQL，
  后期按模块逐步迁移至 Golang（API 契约保持不变，前端无感）。
- 内容同步方向：cn → com（单向只读 CDC），用户数据绝不互通。

#### 站间链接规则

- 两站 Header 均提供指向对方站点的外部链接（新标签页打开）。
- 链接必须指向对方站点首页（`https://xyzidea.cn` 或 `https://xyzidea.com`）。
- 不做深度链接（不跨站路由到特定文章/工具）。

#### 数据隔离红线（最高优先级）

1. cn 用户数据（手机号、收藏等）MUST NOT 流向海外节点，只存于国内 PostgreSQL。
2. 全球站 MUST NOT 收集任何中国敏感个人信息（手机号、微信号、QQ号、
   身份证、生物特征），绝对禁止。
3. 两站用户账号体系完全独立，不做单点登录。
4. 账号注销后 30 天内 MUST 物理删除，CDC 同步清理两边数据。
5. 欧盟用户（com 站）仅提供只读权限，不开放注册。

#### Vite 配置约束

- 校园前端 MUST 使用 `resolve.alias` 显式统一 React 实例路径，
  避免 pnpm 严格模式下嵌套 `node_modules` 导致的 `Invalid hook call`。
- 全球站前端 MUST 使用 `resolve.dedupe` 去重 React 相关依赖。
- 修改任一 Vite 配置后 MUST 清除 `node_modules/.vite` 缓存再重启验证。

#### 域名验证

- `vite.config.campus.ts` 的 `server` MUST NOT 包含全球站域名。
- `vite.config.ts` 的 proxy MUST NOT 包含校园站域名。

**原因**：合规要求（ICP 备案 vs 海外部署）决定了两面 MUST 物理隔离，
同一仓库降低共享代码维护成本但 MUST 严格防止交叉污染。功能一致性确保
产品体验统一，降低开发和维护成本。

## 开发工作流

1. **规格先行**：每个功能先在 `NorthStar/openspec/Changes/` 形成图纸，
   再进入代码实现。无规格不动代码。
2. **宪法关卡**：实现前必须逐条核对上述原则是否合规；违规项必须在计划的
   复杂度追踪表中显式标注。
3. **渐进交付**：按优先级实现用户故事（P1 → P2 → P3），每个故事必须可独立
   测试、独立部署。
4. **双面验证**：共享代码修改后必须同时在两个前端启动验证，确保无交叉污染。
5. **缓存清理**：修改 Vite 配置或安装新依赖后，必须删除
   `node_modules/.vite` 目录再重启 dev server，避免 Outdated Optimize Dep
   错误。
6. **发布审核**：内容变更需审核员通过；技术变更需宪法合规检查。
7. **审计留痕**：所有后台操作、AI 建议、内容版本变更必须记录日志。

## 治理

本宪法高于所有其他开发实践、临时讨论与个人偏好。

**修订流程**：
- 以规格文档形式在 `openspec/Changes/` 提出修订提案。
- 提案必须包含：修订理由、受影响原则、现有代码迁移方案、审批签字。
- MAJOR 版本：原则删除或重新定义（向后不兼容）。
- MINOR 版本：新增原则或实质性扩展。
- PATCH 版本：措辞澄清、排版、笔误修正。

**合规审查**：每次 PR 或评审必须验证宪法合规性。未经论证的复杂度违规必须在合入前解决。

**运行时指引**：日常开发使用 `.specify/` 模板与命令。模板与宪法冲突时，以宪法为准。
