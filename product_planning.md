## 一、用"原型吃掉文档"：核心功能清单 → 可运行原型

### 1.1 核心功能清单（用户问题-解决方案-验收标准）

> [!IMPORTANT]
> Cat Wu 的核心主张：**文档是二等公民，可运行的原型才是一等公民**。PRD 中的每个功能必须能被"点击"、被"体验"、被"吐槽"，否则它就不存在。

| # | 用户问题 | 解决方案 | 验收标准（可测试） | 原型优先级 |
|---|---------|---------|-----------------|----------|
| F1 | "我有个问题，不知道该用什么 AI 工具" | AI 搜索：输入自然语言 → 返回摘要 + 推荐工具 + 推荐文章 | 输入"我想做短视频"→ 3s 内返回 ≥1 工具 + ≥1 文章 + 路径建议 | **P0** |
| F2 | "搜到了工具，但不知道怎么用" | 方案生成：选工具 → 输入目标 → AI 生成 3-5 步操作路径 | 选择 2 个工具 + 输入目标 → 生成可读方案 → 一键保存 | **P0** |
| F3 | "我想把方案带走" | 导出与分享：方案导出为 md/txt/csv + 剪贴板分享 | 点击导出 → 文件下载成功；点击分享 → 剪贴板包含产品名+工具列表 | **P0** |

## 二、3 个"快速实验"方向（1-2 周内可验证）

> [!TIP]
> Cat Wu 的实验哲学：**实验只验证"不确定的假设"**。已确定的需求（如"用户需要搜索"）不需要实验。实验聚焦于"我们不知道的事"。

### 实验 1：AI 搜索结果的"方案转化率"

**不确定假设**：用户在 AI 搜索后，是否真的会点击"加入方案"并生成完整方案？还是搜到答案就走了？

| 维度 | 内容 |
|------|------|
| **目标** | 验证"AI 搜索 → 方案生成"路径的自然转化率 |
| **方法** | 在现有全球站原型中添加行为埋点：记录 `搜索完成` → `点击加入方案` → `点击生成` → `点击保存` 四个节点的转化漏斗 |
| **参与者** | 投放给 50 名种子用户（AI 从业者社群招募） |
| **核心指标** | 搜索→加入方案 转化率（目标 ≥20%）；加入→生成 转化率（目标 ≥50%） |
| **时间** | 1 周（第 1-2 天部署埋点 + 第 3-7 天收集数据） |
| **谁来做** | 开发：添加前端埋点 Hook（`useTrackFunnel`）；运营：社群招募 + 数据汇总 |
| **失败方案** | 若搜索→加入 转化率 <10%：① 在 AI 搜索结果中直接展示"一键生成方案"按钮（减少步骤）；② 在搜索结果卡片中增加"适用场景"标签降低决策成本 |

### 实验 2：校园站 AI 搜索的"内容满意度"

**不确定假设**：用 AI 搜索回答校园生活问题（如"学校附近有什么好吃的"），在内容库较小（30 篇）时，用户是否能获得有用的回答？

| 维度 | 内容 |
|------|------|
| **目标** | 验证校园站 AI 搜索在冷启动期的内容满足率 |
| **方法** | 在校园站 AI 搜索结果底部添加"这个回答有帮助吗？ 👍/👎"按钮；同时记录"无结果搜索词"列表 |
| **参与者** | 投放给 30 名在校学生（线下扫码招募） |
| **核心指标** | 👍 率（目标 ≥60%）；无结果搜索占比（目标 <30%） |
| **时间** | 1 周（第 1 天部署反馈按钮 + 第 2-7 天收集数据） |
| **谁来做** | 开发：添加反馈组件 + 无结果日志；研究员：分析无结果搜索词，补充内容 |
| **失败方案** | 若 👍 率 <40%：① 扩充种子内容至 50 篇/领域；② 优化 AI 系统提示，增加"结合线上公开信息补充回答"的能力；③ 若仍不达标，考虑临时接入外部知识库（如大众点评 API） |

### 实验 3：Topic 关注作为留存机制的有效性

**不确定假设**：用户是否愿意"关注 Topic"并因邮件通知而回访？还是只是一次性消费内容？

| 维度 | 内容 |
|------|------|
| **目标** | 验证 Topic 关注 → 邮件通知 → 回访的留存链路 |
| **方法** | 在文章页添加"关注这个话题"按钮（邮箱订阅）；发布新文章时手动发送邮件通知；追踪邮件打开率和回访率 |
| **参与者** | 从实验 1 和实验 2 的用户中，筛选完成完整路径的活跃用户（预计 20-30 人） |
| **核心指标** | 关注转化率（目标 ≥15%）；邮件打开率（目标 ≥30%）；7 日回访率（目标 ≥15%） |
| **时间** | 2 周（第 1 周部署 + 招募关注者；第 2 周发送通知 + 追踪回访） |
| **谁来做** | 开发：实现关注按钮 + 邮件列表（初期用 Resend）；研究员：撰写新文章触发通知 |
| **失败方案** | 若 7 日回访率 <8%：① 将通知从邮件改为微信服务号推送（cn 站）；② 增加"周刊摘要"代替单篇通知；③ 若关注转化率 <5%，考虑改为"收藏即关注"的隐式关注模式 |

### 实验时间线总览

```
Week 1:  ████████ 实验1（方案转化率） + 实验2（内容满意度）
Week 2:  ████████ 实验1/2 数据分析 + 实验3（Topic 关注）开始
Week 3:  ████████ 实验3 通知发送 + 全部实验结论汇总
Week 4:  → 根据实验结论调整产品路线图
```

## 三、持续交付机制（每周 1 次小版本迭代）

### 3.1 迭代周期设计

> [!IMPORTANT]
> Cat Wu 经验：Claude Code 团队保持"每天都能 ship"的节奏。对于你的产品阶段，**每周五发布小版本** 是最合适的节奏——足够快来保持动力，又不至于因节奏太快而牺牲质量。

```
┌─────────────────────────────────────────────────┐
│  每周迭代周期                                      │
│                                                 │
│  周一  │ 回顾上周数据 + 确定本周优先级              │
│  周二  │ 设计 + 开发（核心功能）                    │
│  周三  │ 开发 + 内部测试                           │
│  周四  │ Bug 修复 + 用户反馈处理 + 发布准备          │
│  周五  │ 发布小版本 + 更新 CHANGELOG                │
│  周末  │ 收集用户反馈 + 撰写新内容                   │
└─────────────────────────────────────────────────┘
```

### 3.2 版本评审标准

每次发布前必须通过以下 4 项检查：

| # | 评审项 | 判定标准 | 否决条件 |
|---|--------|---------|---------|
| 1 | 用户核心问题 | 本次发布是否解决了至少 1 个用户核心问题？ | 纯技术重构不包含用户可感知改进 |
| 2 | 数据支撑 | 功能方向是否有数据或用户反馈支撑？ | "我觉得用户需要"但无数据佐证 |
| 3 | 宪法合规 | 是否通过宪法原则检查（12 条对照）？ | 违反数据隔离红线 / 安全不变量 |
| 4 | 回退方案 | 是否有 Feature Flag 或快速回滚能力？ | 新功能无法通过开关关闭 |

### 3.4 优先级排序：Impact-Effort 矩阵

```
          高 Impact
              │
   ┌──────────┼──────────┐
   │  Quick   │  Big     │
   │  Wins    │  Bets    │
   │  ★★★★★  │  ★★★★   │
   │ (先做)   │ (规划做)  │
   ├──────────┼──────────┤
   │  Fill    │  Money   │
   │  Ins     │  Pit     │
   │  ★★★    │  ★      │
   │ (有空做)  │ (不做)   │
   └──────────┼──────────┘
              │
          低 Impact
    低 Effort ←→ 高 Effort
```

2. 首页标题更新为产品定位文案（0.5 天）
3. 添加 AI 搜索结果反馈按钮（0.5 天）
4. 图片懒加载替换外部图源（0.5 天）

**当前 Big Bets（高 Impact + 高 Effort）**：
1. 后端 API 搭建（5 天）
2. 校园站前端重建（4 天）
3. Topic 关注 + 邮件通知（3 天）

### 3.5 模型迭代应对方案

> [!NOTE]
> AI 模型能力在快速进步。当前使用智谱 GLM-4-Flash，但需要为模型升级做好准备。

```
┌──────────────────────────────────────────────────┐
│  模型迭代响应策略                                    │
│                                                  │
│  当前：智谱 GLM-4-Flash                             │
│                                                  │
│  触发条件 → 响应动作：                                │
│                                                  │
│  1. 新模型上线（如 GLM-5）                           │
│     → AI 网关的 model 参数可配置，后台切换即生效        │
│     → 不需要前端改动（前端只调 /api/ai/search）        │
│                                                  │
│  2. 模型能力大幅提升（支持 RAG / 长上下文）             │
│     → 实验阶段：用 Feature Flag 对 10% 用户灰度       │
│     → 验证阶段：比较新旧模型的 👍 率差异               │
│     → 正式上线：通过后台配置切换所有用户                 │
│                                                  │
│  3. 成本显著降低（如 token 价格降 50%）                │
│     → 提高免费额度（如 3→10 次/天）                   │
│     → 开放更多 AI 功能（如 AI 辅助创作）               │
│                                                  │
│  4. 竞品模型表现更好（如 DeepSeek / Qwen）             │
│     → AI 网关已设计为多模型支持                       │
│     → 可同时接入多个模型，按场景选择最优                 │
│     → A/B 测试对比模型效果                            │
│                                                  │
│  架构保障：                                         │
│     前端 ←→ AI 网关 ←→ 模型厂商 API                  │
│     ↑ 解耦层 ↑  ↑ 可插拔 ↑                          │
│     前端不感知模型变化，只依赖 AI 网关契约              │
└──────────────────────────────────────────────────┘
```

## 五、AI 辅助开发流程

### 5.1 用 AI 生成原型

> [!TIP]
> Cat Wu 的核心实践：Claude Code 团队自己就是用 Claude Code 来开发 Claude Code 的。你也应该用 AI 来开发 AI 产品。

| 场景 | AI 工具 | 具体方法 | 预期效率提升 |
|------|--------|---------|------------|
| **PRD → React 组件** | Claude Code / Gemini | 将 PRD 中的页面描述、交互规则喂给 AI，直接生成 React + Tailwind 页面组件 | 单页面从 4h → 1h |
| **API 契约 → Hono 路由** | Claude Code | 将 `specs/006/contracts/*.md` 中的 API 契约转化为 Hono.js 路由代码 | 路由层从 2h → 30min |
| **Prisma Schema 生成** | Claude Code | 将 PRD §11 数据规格转化为 `schema.prisma` | 数据模型从 2h → 20min |
| **种子内容生成** | 智谱 GLM-4 | 批量生成校园站种子文章（结构化 Markdown） | 10 篇/小时 → 30 篇/小时 |
| **国际化翻译** | 智谱 GLM-4-Flash | 中文文章 → 英文文章（标题+摘要人工审校） | 翻译 1 篇从 1h → 10min |
| **UI 微调** | Claude Code + Tailwind | 描述"间距再大一点""按钮颜色改为品牌色"→ 直接修改代码 | 样式调整从 30min → 5min |

### 5.2 用 AI 测试原型

| 测试类型 | AI 辅助方法 | 工具 |
|---------|------------|------|
| **功能测试** | 用 Claude Code 生成 Playwright E2E 测试脚本，覆盖核心路径 | Playwright + Claude Code |
| **异常路径** | 让 AI 枚举"用户可能的异常操作"（空输入、超长输入、特殊字符），生成测试用例 | Claude Code 生成 Vitest 用例 |
| **可用性模拟** | 描述用户场景（如"一个大一新生想找学校附近的奶茶店"），让 AI 模拟用户操作路径，找出卡点 | Claude Code 场景分析 |
| **内容质量** | 批量生成 100 个搜索查询，测试 AI 搜索的命中率和回答质量 | 脚本自动化 + 人工抽检 |
| **性能压测** | 用 AI 生成 k6/Artillery 脚本，模拟并发 AI 搜索请求 | k6 + Claude Code |

## 七、关键决策点与风险预案

### 7.1 第 4 周决策点

基于 3 个实验结果，需要在第 4 周做出以下决策：

| 决策项 | 若实验成功 | 若实验失败 |
|--------|----------|----------|
| 方案生成路径 | 保持当前"搜索→加入→生成"三步路径 | 简化为"搜索→一键生成"两步，减少摩擦 |
| 校园站内容策略 | 继续自建内容，每领域 ≥30 篇 | 接入外部数据源（大众点评 / 小红书公开数据） |
| 留存机制 | 全面推广 Topic 关注 + 邮件通知 | 转向"搜索历史"或"周刊推送"替代方案 |

### 7.2 风险预案

| 风险 | 概率 | 影响 | 预案 |
|------|------|------|------|
| AI 搜索成本失控 | 中 | 高 | AI 网关分层限流 + 全局日上限 + 异常 IP 拦截 |
| 校园站内容不足 | 高 | 中 | 每领域 <10 篇时使用 Locked 态；AI 辅助批量生成种子内容 |
| 种子用户招不到 | 中 | 中 | 线下扫码（咖啡券激励）+ AI 社群发布 + Product Hunt 发布 |
| 智谱 API 不稳定 | 低 | 高 | Fallback 到演示模式（已实现）+ AI 网关可切换模型 |
| 验证假设全部失败 | 低 | 高 | 回到"内容驱动"策略，先做好 SEO + 内容质量，再迭代产品功能 |
